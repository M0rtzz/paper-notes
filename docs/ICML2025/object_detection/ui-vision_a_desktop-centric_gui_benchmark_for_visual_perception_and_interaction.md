# UI-Vision: A Desktop-centric GUI Benchmark for Visual Perception and Interaction

**会议**: ICML 2025  
**arXiv**: [2503.15661](https://arxiv.org/abs/2503.15661)  
**领域**: 目标检测  
**关键词**: GUI 基准, 桌面环境, 元素定位, 布局理解, 动作预测, 视觉感知, 自主智能体

## 一句话总结

提出 UI-Vision，首个面向桌面环境的许可开放、离线评估基准，覆盖 83 个软件应用，定义了元素定位、布局定位和动作预测三个由细到粗的评估任务，揭示了 SOTA 模型（如 UI-TARS-72B）在专业软件理解和空间推理上的关键局限。

## 研究背景与动机

自主 GUI 智能体通过导航图形用户界面自动化文档编辑、文件管理等任务，极大提升计算机工作流效率。然而当前研究存在以下问题：

**在线 vs 离线**：现有基准多为在线交互式，难以标准化评估，且复现成本高

**桌面环境缺失**：大多数基准聚焦 Web 和移动端，桌面环境因数据采集困难和许可问题被严重忽视

**标注稀疏**：现有桌面数据集缺乏密集标注（bounding boxes、UI 标签、完整操作轨迹）

**评估粒度粗糙**：缺乏从细粒度（元素级）到粗粒度（动作级）的系统化评估任务

UI-Vision 的目标是填补这些空白，提供高质量、许可友好的离线基准。

## 方法详解

### 整体框架

UI-Vision 基准包含三个核心组件：

1. **数据集**：来自 83 个桌面软件应用的人类操作示范，包含密集标注
2. **三级评估任务**：由细到粗的层次化任务设计
3. **标准化评估指标**：每个任务配有明确定义的度量标准

### 数据标注体系

数据集提供以下类型的标注：
- **Bounding boxes**：UI 元素的精确边界框
- **UI 标签**：每个界面元素的功能描述
- **操作轨迹**：完整的人类交互序列（点击、拖拽、键盘输入）

数据涵盖多种专业软件类别，包括创意工具、开发环境、办公套件等，总计约 1464 条标注样本。

### 三个评估任务

**任务 1：Element Grounding（元素定位）**

给定屏幕截图和元素描述，模型需要预测目标 UI 元素的 bounding box。此任务包含三个子集：
- **Basic**：基础元素识别（按钮、文本框等）
- **Functional**：基于功能描述的元素定位
- **Spatial**：涉及空间关系推理的元素定位

评估指标：IoU（Intersection over Union）匹配准确率。

**任务 2：Layout Grounding（布局定位）**

给定屏幕截图，模型需要识别和定位界面中的所有 UI 元素及其层次关系。评估模型对整体布局的理解能力。

**任务 3：Action Prediction（动作预测）**

给定屏幕截图和任务描述，模型需要预测下一步操作（点击位置、拖拽轨迹或键盘输入）。这是最具挑战性的端到端任务。

### 评估设计

所有评估采用离线模式，不需要与实际操作系统交互，确保：
- 可复现性：每次评估基于相同的静态截图
- 标准化：统一的指标定义和评估流程
- 可扩展性：支持新模型的快速评估

## 实验关键数据

### 主实验：跨模型评估对比

| 模型 | Element Grounding (Basic) | Element Grounding (Functional) | Element Grounding (Spatial) | Layout Grounding | Action Prediction |
|------|--------------------------|-------------------------------|---------------------------|-----------------|-------------------|
| GPT-4o | 中等 | 较低 | 较低 | 较低 | 较低 |
| Claude 3.5 | 中等 | 中等 | 较低 | 中等 | 较低 |
| UI-TARS-72B | 最高 | 较高 | 中等 | 较高 | 中等 |
| Qwen2-VL | 中等 | 中等 | 较低 | 中等 | 较低 |

**关键发现**：即便是最强的 UI-TARS-72B，在空间推理和复杂动作（如拖拽）上仍表现不佳。

### 按任务维度的关键发现

| 挑战维度 | 表现特征 | SOTA 模型典型错误 |
|---------|---------|-------------------|
| 专业软件理解 | 所有模型表现显著下降 | 无法识别专业工具栏元素 |
| 空间推理 | Spatial 子集准确率大幅低于 Basic | 方位关系判断错误 |
| 拖拽操作 | 动作预测任务中最低分 | 起终点坐标预测偏差大 |
| 键盘输入 | 次于点击但优于拖拽 | 快捷键组合预测困难 |
| 跨应用泛化 | 常见应用 > 专业应用 | 对小众软件界面理解差 |

### 软件类别覆盖

基准涵盖 83 个桌面软件应用，跨越多个类别：
- 创意/设计工具（如图像编辑、视频编辑）
- 开发工具（如 IDE、终端）
- 办公/生产力（如文档、表格）
- 系统工具（如文件管理器、设置）
- 通信工具（如邮件客户端）

## 亮点与洞察

1. **首个桌面离线基准**：填补了 GUI 智能体评估的重要空白，不依赖在线交互环境
2. **三级任务设计精巧**：从元素级到动作级的层次化任务，能精确定位模型的能力瓶颈
3. **许可友好**：MIT 许可证，所有数据和标注完全开放，促进社区研究
4. **揭示关键短板**：发现即使 72B 规模的专用模型也在桌面专业软件上表现不佳，指明了重要研究方向
5. **密集标注价值**：bounding box + 功能标签 + 操作轨迹的组合为多维度评估提供基础

## 局限性

1. **数据规模有限**：约 1464 条标注，相比 Web 基准（如 Mind2Web）数量较少
2. **操作系统覆盖不全**：主要针对特定桌面操作系统，缺乏跨 OS 的泛化评估
3. **静态截图局限**：离线评估无法捕捉动态交互中的时序依赖和反馈循环
4. **缺乏真实任务完成评估**：仅评估单步预测，未衡量多步骤任务的端到端成功率
5. **标注者主观性**：bounding box 标注和功能描述可能存在标注者偏差

## 相关工作与启发

- **与 ScreenSpot/OSWorld 的关系**：ScreenSpot 关注移动端，OSWorld 关注在线交互，UI-Vision 关注桌面离线评估，三者互补
- **与 Mind2Web 的差异**：Mind2Web 面向 Web，UI-Vision 面向桌面原生应用，后者界面更复杂多样
- **启发**：桌面 GUI 的视觉复杂性（专业工具栏、多层菜单）对现有 VLM 提出了独特挑战，值得在训练数据中增加此类样本
