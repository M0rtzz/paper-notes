# Can One Safety Loop Guard Them All? Agentic Guard Rails for Federated Computing

**会议**: ICML 2025  
**arXiv**: [2506.20000](https://arxiv.org/abs/2506.20000)  
**代码**: 无  
**领域**: AI安全  
**关键词**: 联邦计算, 隐私保护, 安全框架, Agentic AI, 护栏机制

## 一句话总结
提出 Guardian-FC——一个双层框架，通过 Agentic-AI 控制平面和后端无关的领域特定语言（DSL），统一管理不同隐私机制（FHE/MPC/DP）的安全护栏，实现联邦计算中的跨后端一致性安全执行和可审计性。

## 研究背景与动机

**领域现状**：联邦计算结合 FHE、MPC、DP 等多种隐私技术实现协作分析，但每种技术有各自独立的安全指标和监控机制（如 CKKS 跟踪噪声预算，DP 跟踪 ε 预算，MPC 跟踪份额完整性）。
**现有痛点**：安全框架碎片化——需要为每种后端开发独立的监控/控制/审计工具；多后端组合工作流缺乏端到端安全验证；不一致的安全策略增加隐私泄露风险。
**核心矛盾**：异构隐私后端各有不同的安全概念和度量，难以用统一框架管理。
**本文要解决什么**：如何构建一个可复用的、后端无关的安全框架，统一感知、预测和响应安全风险？
**切入角度**：将护栏逻辑与隐私机制解耦——通过 DSL 抽象计算操作，通过可插拔执行提供者适配不同后端。
**核心 idea**：双层架构——上层 Agentic-AI 控制平面执行有限状态安全循环，下层联邦计算层通过 DSL 插件和可互换执行提供者运行计算。

## 方法详解

### 整体框架
Guardian-FC 由两层组成：
- **控制平面（Agentic-AI 层）**：处理签名元数据（不接触原始数据），通过遥测收集器→哨兵引擎→命令分发器→审计引擎实现 Sense→Predict→Act→Prove 安全循环
- **数据平面（联邦计算层）**：多节点 + 中央聚合器，运行用后端无关 DSL 编写的插件，通过可互换的执行提供者（EP）绑定到具体隐私后端

### 关键设计

1. **后端无关 DSL**:

    - 做什么：提供一种统一语言描述联邦计算工作流，独立于具体隐私技术
    - 核心思路：DSL 定义抽象操作（如 encrypt、aggregate、add_noise），执行提供者将这些操作映射到具体后端实现
    - 设计动机：新后端只需实现 EP 接口，无需修改控制平面或护栏逻辑

2. **有限状态安全循环**:

    - 做什么：对每个作业的生命周期进行有限状态机管理
    - 核心思路：状态包括 Registered→Running→Completed/Aborted，控制平面持续评估安全谓词（如噪声预算是否耗尽、ε 是否溢出），违反时触发护栏命令
    - 设计动机：FSM 模型支持形式化验证，确保安全属性可证

3. **Manifest 注册协议**:

    - 做什么：作业提交时进行准入检查
    - 核心思路：JSON manifest 指定插件、执行提供者、DSL 操作和护栏谓词，系统在执行前验证兼容性
    - 设计动机：fail-fast 策略，防止不兼容配置浪费计算资源

### 损失函数 / 训练策略
- 无训练过程，纯框架/系统设计论文
- 提供形式化 FSM 模型和安全谓词定义

## 实验关键数据

### 主实验
本文为框架设计论文，**无定量实验**。提供了三个定性场景：

| 场景 | 隐私后端 | 安全风险 | Guardian-FC 响应 |
|------|---------|---------|-----------------|
| CKKS 噪声耗尽 | FHE | 噪声预算接近耗尽 | 哨兵预警→命令中止/重引导 |
| DP ε 预算溢出 | DP | 累积 ε 超过阈值 | 安全循环触发作业终止 |
| MPC 份额完整性异常 | MPC | 份额验证失败 | 检测异常→隔离节点 |

### 关键发现
- 统一安全循环可以跨后端一致地检测和应对安全风险
- Manifest 准入检查避免配置不兼容问题
- 审计引擎的 Merkle 账本确保操作不可篡改

## 亮点与洞察
- **控制平面与数据平面解耦**的架构设计清晰，后端扩展性好（加新后端只需写 EP）
- Agentic-AI 的安全循环概念有趣——将 Agent 的感知-决策-行动循环用于系统安全管理
- 提出了明确的研究议程（自适应护栏调优、多后端组合、DSL 编译器等）

## 局限性 / 可改进方向
- **无实际实现和定量评估**——纯框架设计+定性分析，缺乏性能数据
- DSL 的具体规范尚未定义，仅描述了概念
- 未讨论控制平面自身的安全性（如控制平面被攻击怎么办）
- Agentic-AI 组件的具体实现（如预测模型）未展开
- 实际部署中的开销（如签名遥测的延迟、Merkle 账本的存储）未量化

## 相关工作与启发
- **vs 现有联邦学习框架**: Flower、PySyft 等专注于 ML 训练，Guardian-FC 聚焦安全管理层
- **vs 单后端安全方案**: 各后端独立的安全工具无法统一管理，Guardian-FC 提供跨后端视角
