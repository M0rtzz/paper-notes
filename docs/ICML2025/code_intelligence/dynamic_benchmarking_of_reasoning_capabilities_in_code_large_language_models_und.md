# DyCodeEval: Dynamic Benchmarking of Reasoning in Code LLMs Under Data Contamination

**会议**: ICML2025  
**arXiv**: [2503.04149](https://arxiv.org/abs/2503.04149)  
**代码**: [项目页](https://codekaleidoscope.github.io/dycodeeval.html)  
**领域**: llm_nlp  
**关键词**: 代码LLM评估, 数据污染, 动态基准, 蜕变测试, 语义等价

## 一句话总结
提出DyCodeEval——用多Agent系统自动生成编程问题的语义等价变体来动态评估Code LLM，修改问题的上下文描述（不改变核心算法逻辑）以避免记忆化，在18个Code LLM上验证其有效反映真实推理能力。

## 研究背景与动机

### 数据污染问题
Code LLM训练时不可避免地见过HumanEval等静态基准。在污染环境下评估不能反映真实推理能力——只反映记忆化。

### 已有解决方案的不足
- LiveCodeBench从在线平台抓新题：仍需人工出题
- PPM手动定义算子：语义复杂度不可控，难以归因性能下降

### DyCodeEval的创新
受蜕变测试启发：修改问题的"复杂度无关部分"（上下文描述）而保持"复杂度相关部分"（算法逻辑），生成语义等价但文本不同的变体。

## 方法详解

### 蜕变关系定义
编程问题 = 算法抽象（决定难度）+ 上下文描述（不影响难度）
修改上下文 → 语义变化但难度不变

### 多Agent生成
LLM Agent协作生成：
1. 提取核心算法逻辑
2. 生成新的上下文描述
3. 验证等价性和可解性

### 验证Agent
概率性预言机验证新生成问题的正确性和一致性。

### 评估框架
在原始题+变体题上评估同一LLM，对比性能差异→揭示记忆化程度。

## 实验关键数据

### 18个Code LLM评估

| LLM类别 | 原始基准准确率 | DyCodeEval准确率 | 差异 |
|---------|-------------|----------------|------|
| 闭源(GPT-4等) | 高 | 中-高 | 小 |
| 开源大模型 | 中-高 | 中 | 中等 |
| 开源小模型 | 中 | 低 | **大(=污染)** |

### 手动污染验证

| 设置 | 效果 |
|------|------|
| 在原题上微调 | 原题准确率大幅提升 |
| 在DyCodeEval上测试 | **准确率未提升** |

### 关键发现
1. 小模型的原始基准vs DyCodeEval差距最大——暗示记忆化更严重
2. 大模型差距较小——说明它们确实学到了推理能力
3. 上下文修改有效避免了表面记忆
4. 验证Agent确保了变体的正确性
5. 两个种子数据集上结论一致

## 亮点与洞察

1. 蜕变测试思想的巧妙迁移——从软件测试到LLM评估。
2. 区分了"复杂度相关"和"复杂度无关"部分——理论基础清晰。
3. 自动化程度高——不需要人工出新题。
4. 验证Agent保证了变体质量——避免引入新的不公平。
5. 18个LLM的大规模评估增强了结论可信度。

## 局限性 / 可改进方向

1. 上下文修改可能对某些算法不完全等价（如需要领域特定知识的题）。
2. 验证Agent本身可能有盲区（概率性预言机）。
3. 仅在Python题目上验证。
4. 与LiveCodeBench的直接对比不够详细。
5. 未考虑chain-of-thought推理下的污染效果。

## 相关工作与启发

- 与LiveCodeBench的区别：LCB靠时间戳过滤，DyCodeEval靠语义变换。
- 与PPM的区别：PPM手动定义算子，DyCodeEval自动化。
- 启发：蜕变测试可推广到NLP/数学等其他LLM基准的防污染。

## 补充

### 蛀变关系的具体示例
原题：“雨打在窗户上，求滤波后信号”→变体：“电磁波穿过介质，求滤波后信号”。算法完全相同(都是滤波器)，但上下文不同。

### 小模型偏差大的解释
小模型可能更依赖表面模式匹配而非理解算法逻辑，因此上下文变化就让它们失败。

### 验证Agent的作用
确保变体问题的正确性：检查测试用例能否通过、解题逻辑是否一致。
