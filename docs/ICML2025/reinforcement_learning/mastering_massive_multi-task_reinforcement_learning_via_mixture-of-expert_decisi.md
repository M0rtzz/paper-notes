# Mastering Massive Multi-Task Reinforcement Learning via Mixture-of-Expert Decision Transformer

**会议**: ICML2025  
**arXiv**: [2505.24378](https://arxiv.org/abs/2505.24378)  
**代码**: 待确认  
**领域**: reinforcement_learning  
**关键词**: 多任务强化学习, Decision Transformer, Mixture-of-Experts, 任务可扩展性, 参数可扩展性

## 一句话总结

提出 M3DT 框架，将 Mixture-of-Experts (MoE) 引入 Decision Transformer 架构，通过任务分组让每个专家只学习少量任务的特定知识，并设计三阶段训练机制（骨干→专家→路由器）依次优化各模块，成功将离线多任务 RL 扩展到 160 个仿真控制任务且性能持续提升。

## 研究背景与动机

1. **领域现状**：近年来 Decision Transformer (DT) 及 Trajectory Transformer 等方法将离线 RL 重新建模为序列建模问题，展现了从大规模数据集中训练决策智能体的能力。这类高容量序列模型也被应用于多任务 RL (MTRL)，期望借鉴 LLM 在海量任务上的泛化能力。

2. **现有痛点**：
   - **任务可扩展性不足**：大多数现有工作仅在 Atari 或 Meta-World 上处理几十个任务（如 Lee et al., 2022; Hu et al., 2024），扩展到更多任务后性能显著下降。Gato (Reed et al., 2022) 虽然覆盖 600+ 任务，但在控制任务上表现不佳。
   - **参数扩展效率低**：简单增大模型尺寸会迅速触及性能天花板。Gato 和 Multi-Game DT 虽然实验表明增大模型有提升，但模型尺寸增长与性能增益不成比例，不是高效的参数扩展方式。

3. **核心矛盾**：随着任务数增加，所有任务共享同一组参数导致梯度冲突加剧、性能退化；而单纯增大共享参数并不能有效缓解这一问题。

4. **本文要解决什么**：如何在任务数量极大（160 个）时，既保持每个任务的学习质量，又实现高效的参数扩展？

5. **切入角度**：作者通过实证分析发现两个关键洞察：
   - 性能退化在任务数较少时最为剧烈，任务数足够多后退化趋于平缓 → **反过来看，减少每个参数子集需要学习的任务数可以显著提升性能**
   - 单纯扩大共享参数很快触顶 → **扩展参数的同时减少任务负载，才能获得最显著的性能增益**

6. **核心 idea 一句话**：用 MoE 实现"参数分离 + 任务分组"，每个专家只负责一个小任务子集，增加专家数量既扩展参数又降低每个专家的任务负载，双重增益。

## 方法详解

### 整体框架

M3DT 的整体 pipeline 基于 Decision Transformer 骨干网络，在其上引入 MoE 层实现参数分离：

- **输入**：与标准 DT 一致，将离线数据集中的轨迹 $(R_1, s_1, a_1, R_2, s_2, a_2, \ldots)$ 作为序列输入，其中 $R_t$ 为 return-to-go、$s_t$ 为状态、$a_t$ 为动作
- **骨干网络**：共享的 Transformer 层学习跨任务的通用知识
- **MoE 层**：每个 Transformer block 中 FFN 替换为 MoE 结构，每个专家 FFN 被分配一组特定任务子集
- **路由器**：基于任务 ID 将输入路由到对应专家
- **输出**：预测下一步动作 $a_t$

整个框架的核心设计思想是：**共享骨干学通用表征，任务分组后各专家只学少量任务的特定策略，通过增加专家数同时实现参数扩展和任务负载降低**。

### 关键设计

1. **MoE 增强的 Decision Transformer 架构**
   - 做什么：在 DT 的每个 Transformer block 中，将标准 FFN 替换为多个并行的专家 FFN
   - 核心思路：每个专家 $E_k$ 负责学习一组任务子集 $\mathcal{T}_k$ 的特定知识。对于输入 token $x$，路由器根据其所属任务 ID 选择对应专家进行前馈：$\text{MoE}(x) = E_{r(x)}(x)$，其中 $r(\cdot)$ 是路由函数。共享的 attention 层仍然处理所有任务，学习跨任务的通用时序关系
   - 设计动机：相比让所有参数共享所有任务，MoE 通过参数分离大幅减少每个专家面对的任务数量。根据作者的分析，减少任务数可以显著缓解梯度冲突并提升性能。同时增加专家数是最高效的参数扩展方式（Fedus et al., 2022），引入大量参数但每个前馈只激活一个专家，计算成本不随专家数线性增长

2. **任务分组 (Task Grouping)**
   - 做什么：将 160 个任务显式地划分为若干组，每组分配给一个专家
   - 核心思路：采用确定性的任务到专家映射，而非标准 MoE 中基于 softmax 的动态路由。具体地，将任务按领域/相似性进行分组，相似的任务被分到同一个专家，使专家内部学习的知识具有**局部一致性**，减少组内梯度冲突
   - 设计动机：标准 MoE 的动态路由（如 top-k gating）在 RL 场景中面临训练不稳定问题——RL 数据分布和标准 NLP 差异很大，轨迹 token 的路由分配难以收敛。显式分组避免了路由的不稳定性，同时利用任务相似性使专家专精化更高效

3. **三阶段训练机制 (Three-Stage Training)**
   - 做什么：将训练过程分为三个阶段，依次优化不同模块，避免梯度干扰
   - 核心思路：
     - **阶段 1 — 骨干预训练**：冻结所有专家和路由器，仅训练共享 Transformer 骨干（attention 层 + embedding），让骨干学习跨所有任务的通用表征和时序关系
     - **阶段 2 — 专家训练**：冻结骨干参数，为每个专家只用其分配的任务子集数据进行训练，让每个专家学习特定任务组的策略知识
     - **阶段 3 — 路由器微调**：冻结骨干和专家，训练路由器学习如何在推理时将新任务合理分配给最合适的专家
   - 设计动机：同时训练所有模块会导致严重的梯度冲突——骨干需要学通用知识、专家需要学特定知识，两者的梯度方向可能矛盾。分阶段训练让每个模块在无干扰的环境下学习各自的目标，显著降低了 MoE 训练的难度。这与"先预训练后微调"的范式一脉相承

### 损失函数 / 训练策略

- 核心损失与标准 DT 一致，采用动作预测的 MSE 损失：$\mathcal{L} = \mathbb{E}[\|a_t - \hat{a}_t\|^2]$
- 三个阶段分别使用：
  - 阶段 1：全任务数据训练骨干
  - 阶段 2：每个专家独立使用其任务子集数据训练
  - 阶段 3：混合所有任务数据训练路由器
- 不使用辅助负载均衡损失（因为是显式任务分组，无需额外平衡）

## 实验关键数据

### 主实验：不同任务规模下的性能对比

实验在 Meta-World 和 DeepMind Control Suite 等仿真环境中，覆盖 40/80/120/160 个任务规模。

| 任务数 | 方法 | 归一化分数 | 备注 |
|--------|------|-----------|------|
| 40 tasks | DT (baseline) | ~0.65 | 标准 DT 骨干 |
| 40 tasks | M3DT (8 experts) | **~0.82** | 显著提升 |
| 80 tasks | DT (baseline) | ~0.55 | 性能明显退化 |
| 80 tasks | M3DT (8 experts) | **~0.76** | 退化幅度小 |
| 120 tasks | DT (baseline) | ~0.48 | 继续退化 |
| 120 tasks | M3DT (16 experts) | **~0.73** | 仍保持高水平 |
| 160 tasks | DT (baseline) | ~0.42 | 严重退化 |
| 160 tasks | M3DT (16 experts) | **~0.71** | 成功扩展到 160 任务 |

M3DT 在所有任务规模上均大幅超过 DT baseline，且随着任务数增加性能退化远小于 baseline。同时对比 Multi-Game DT、Gato 等方法也有显著优势。

### 消融实验

| 配置 | 归一化分数 (160 tasks) | 说明 |
|------|----------------------|------|
| M3DT (Full) | **~0.71** | 完整模型 |
| w/o MoE (单一大 FFN) | ~0.50 | 去掉 MoE 退化严重，说明参数分离关键 |
| w/o 三阶段训练 (端到端) | ~0.58 | 端到端训练梯度冲突导致性能下降 |
| w/o 任务分组 (随机分配) | ~0.62 | 随机分配不如按相似性分组 |
| 减少专家数 (4 experts) | ~0.64 | 专家数不足，任务负载仍较重 |
| 增加专家数 (32 experts) | ~0.72 | 继续提升但边际收益递减 |

### 关键发现

- **MoE 是核心贡献**：去掉 MoE 后性能从 0.71 降至 0.50，退化幅度最大，证明参数分离对大规模多任务是关键设计
- **三阶段训练很重要**：端到端训练比分阶段训练低 13 个点，说明分阶段训练有效缓解了梯度冲突
- **专家数与性能正相关**：从 4 到 16 专家性能稳步提升，证明了参数可扩展性；32 专家继续微升但收益递减
- **任务分组优于随机分配**：基于相似性的任务分组比随机分配高 9 个点，说明利用任务结构信息有显著收益
- **参数规模 vs 任务规模**：在固定任务数下增加专家可持续提升性能（参数可扩展性），在增加任务数时增加专家可抵消退化（任务可扩展性），两者相互增强

## 亮点与洞察

- **"减少任务数"的逆向思维**：作者从"任务数增加→性能退化"的曲线中发现退化主要集中在小任务数阶段，由此反推出核心策略不是让模型更大，而是让每个参数子集看到更少的任务。这个洞察非常精辟，可以迁移到其他多任务学习场景。
- **MoE 在 RL 中的创新应用**：将 NLP 领域的 MoE 范式引入离线 RL，但做了关键适配——用确定性任务分组替代动态路由，避免了 RL 场景下路由不稳定的问题。这个适配策略对其他需要 MoE 的非 NLP 领域也有参考价值。
- **三阶段训练与梯度冲突的解耦**：多任务 RL 中梯度冲突是核心挑战之一，分阶段训练的思路简洁有效，类似于"先学通用再学专用"的课程设计，可以推广到其他多任务/多目标优化场景。

## 局限性 / 可改进方向

- **仿真环境局限**：所有实验在 Meta-World / DMC 等仿真环境中进行，未验证在真实机器人或更复杂视觉环境中的效果。状态空间均为低维连续向量，面对高维视觉输入可能需要额外设计。
- **任务分组依赖先验知识**：当前任务分组基于领域/相似性的先验划分，面对完全未知的任务集合时如何自动分组是开放问题。可以考虑基于梯度相似性或 task embedding 的自动分组方法。
- **三阶段训练的效率**：分三阶段训练增加了总训练时间，且阶段间的超参（如各阶段训练轮次、学习率）需要额外调优。能否用更优雅的方式（如自适应切换）替代硬性的阶段划分值得探索。
- **动态路由 vs 显式分组**：显式分组虽然稳定，但失去了 MoE 动态路由的灵活性。未来可以在训练稳定后逐步开放路由的自由度，实现"先显式后动态"的过渡方案。
- **缺少在线 RL 验证**：所有实验基于离线数据集，未验证框架在在线 MTRL 或 online fine-tuning 场景中的效果。

## 相关工作与启发

- **vs Multi-Game DT (Lee et al., 2022)**：Multi-Game DT 直接扩大模型参数来处理多任务 Atari，但作者证明了这种朴素扩展效率低、很快触顶。M3DT 通过 MoE 实现更高效的参数利用，同等参数量下性能显著更优。
- **vs Gato (Reed et al., 2022)**：Gato 是通用多模态智能体，覆盖 600+ 任务但控制任务性能不佳。M3DT 聚焦控制任务的 160 个场景，通过任务分组和专家专精化在控制任务上取得更好表现。两者定位不同但 M3DT 的 MoE 思路可以被整合到 Gato 类通用框架中。
- **vs Prompt-DT (Xu et al., 2022)**：Prompt-DT 用 prompt 来区分任务，但模型参数仍完全共享，面对大量任务时梯度冲突问题严重。M3DT 的参数分离是更根本的解决方案。
- **MoE in NLP (Fedus et al., 2022; Lepikhin et al., 2021)**：NLP 领域的 Switch Transformer / GShard 使用动态 top-k 路由，M3DT 的关键区别是用任务 ID 做确定性路由，这是对 RL 场景的重要适配。
