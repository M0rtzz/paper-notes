# UI-Evol: Automatic Knowledge Evolving for Computer Use Agents

**会议**: ICML 2025  
**arXiv**: [2505.21964](https://arxiv.org/abs/2505.21964)  
**作者**: Ziyun Zhang, Xinyi Liu, Xiaoyi Zhang, Jun Wang, Gang Chen, Yan Lu  
**机构**: Microsoft Research Asia  
**领域**: Agent / GUI Agent  
**关键词**: 计算机使用代理, 知识进化, GUI交互, RAG, 自我改进, OSWorld

## 一句话总结

提出 **UI-Evol**，一个即插即用的知识进化模块，通过 **Retrace（回溯）** 和 **Critique（批评）** 两阶段从智能体的实际环境交互中自动修正检索到的外部知识，弥合"知识正确但执行失败"的 knowledge-execution gap，在 OSWorld 上显著提升成功率并大幅降低行为方差。

## 研究背景与动机

### 问题定义

计算机使用代理（Computer Use Agent）需要与 GUI 交互完成复杂长链任务。当前 SOTA 方法（如 Agent S2）采用 RAG 从网页检索外部知识作为任务规划的软先验。但作者发现一个关键矛盾：

> 即使 **90% 的检索知识被人类判定为正确**，最佳代理的成功率也仅为 **41%**。

这说明"知识正确 ≠ 知识可用"。作者将此称为 **Knowledge-Execution Gap**。

### 问题根因分析

实际检索知识的失效原因有三类：

1. **遗漏必要中间步骤**：人类认为"自然"的操作被省略，agent 无法补全
2. **假设与初始条件不一致**：知识假设的系统状态与实际不符
3. **建议过于复杂的操作路径**：如建议鼠标拖拽选中文本，对 agent 而言极难精准执行（Ctrl+A 更可靠）

### 与已有工作的区别

- Monolithic Agent（如 UI-TARS、CogAgent）：端到端单模型直接操作，不使用外部知识
- Modular Agent（如 Agent S/S2、OSCAR）：引入外部工具/知识但缺乏知识校正机制
- Self-Evolution（如 VOYAGER、Self-Refine）：依赖主观反思，而非从视觉证据提取客观行为序列

UI-Evol 的关键创新在于：**不依赖 agent 的主观输出，而是通过截图对比提取客观行为序列，再用结构化推理修正知识**。

## 方法详解

### 整体流程

给定任务指令 + 初始外部知识 → Agent 执行 → 得到交互轨迹（截图序列）→ **Retrace Stage** → **Critique Stage** → 进化后知识写回知识库 → 下一轮执行使用进化知识。

### Stage 1: Retrace（回溯阶段）

**目标**：从截图序列中恢复 agent 的**客观行动序列**（objective action sequence），消除主观行动序列中的幻觉和无效操作。

**核心思路**：

Agent 实际输出的"主观行动序列"可能与真实 UI 状态变化不一致（LMM 幻觉、操作未生效等）。因此 Retrace 阶段不信任 agent 的报告，而是从感知层面重建真实行为。

对于轨迹中的每一步 $t$：
- 输入：操作前截图 $O_t$ 和操作后截图 $O_{t+1}$
- LMM 分析两帧截图的差异，推断实际发生的操作 $\hat{A}_t$
- 若两帧无变化，则 $\hat{A}_t = \text{null}$（操作无效）
- 所有 $\hat{A}_t$ 的联合构成**客观行动序列**

**设计优势**：
- 排除了无效操作和幻觉行为的噪声
- 为后续 Critique 提供可靠的事实依据

### Stage 2: Critique（批评阶段）

**目标**：以外部知识为参考锚，对比客观行动序列，通过 Chain-of-Thought 多步推理修正知识。

四步推理链设计：

**Step 1 — Completion Assessment（完成度评估）**：
判断任务是否成功完成。将客观行动序列的结果与任务目标对比，明确成功/部分完成/失败。

**Step 2 — Deviation Detection（偏差检测）**：
将客观行动序列与原始知识指导的计划进行逐步对比，识别差异并推断偏差的根因。例如分类为"输出/屏幕误解"、"操作粒度不匹配"等。

**Step 3 — Alternative Exploration（替代方案探索）**：
分析 agent 是否在执行中尝试了知识计划之外的有效策略。挖掘这些替代方案可以丰富知识库的多样性。

**Step 4 — Mitigation with Rationales（附理由的修正）**：
综合前三步分析，生成新版知识。修正内容附带因果解释，输出格式与原始知识保持一致（子任务列表形式）。

### 知识格式

采用 Agent S2 的格式：将任务分解为结构化子任务列表（sub-task list）。进化后的知识替换原始知识写回知识库，供后续执行使用。

## 训练与实验设置

### 基准与环境

- **基准**：OSWorld（369 个开放式计算机任务，涵盖 OS、Daily、Office、Professional、Workflow 五大类）
- **最大步数**：15 步
- **基线系统**：Agent S2（OSWorld 当时 SOTA）

### 骨干模型选择

- Retrace 和 Agent 执行：GPT-4o
- Critique 阶段：OpenAI-o3（推理能力更强）
- 为消除 web 搜索随机性，冻结了整个 Perplexica 知识快照
- 温度设为 0，所有可控超参固定

### 并行评测框架

原始 OSWorld 单机评测需约 10 小时。作者在 Azure 上构建了并行评测架构：

- 每个 Azure 实例分配一个虚拟机
- 任务均匀分配到 30 个实例并行执行
- 运行时间从 ~10h 降至 ~2.5h（约 4 倍加速）
- 绕过了 Windows Agent Arena 中反复构建 Docker 镜像的开销

### 稳定性评估

每个实验重复 3 次，报告均值和标准差。即使固定所有超参和知识快照，重复评测结果仍有波动——这一现象此前被研究社区忽视。

## 主要结果

### 核心性能对比（Table 1）

| 方法 | 骨干模型 | 平均成功率 | 标准差 |
|------|----------|-----------|--------|
| Agent S2* (复现) | GPT-4o | 19.5% | ±1.00 |
| **+ UI-Evol** | GPT-4o | **22.0%** | **±0.71** |
| Agent S2* | OpenAI-o3 | 25.6% | ±1.09 |
| **+ UI-Evol** | OpenAI-o3 | **28.4%** | **±0.26** |

关键发现：
- UI-Evol 在两个骨干模型上**均提升成功率并降低标准差**
- o3 + UI-Evol 的标准差仅 0.26，较 GPT-4o 降低约 4.19 倍
- **推理能力更强的模型更能从进化知识中获益**

### 消融：轨迹选择策略（Table 2）

| 选择策略 | 筛选成功率 (SSR) | 平均成功率 |
|---------|----------------|-----------|
| 随机选择 | 70% | 22.0% |
| 基于完成度选择 | 85% | 22.7% |

SSR 定义：$\text{SSR} = \frac{N_{\text{succ}}}{N_{\text{solv}}}$，即选中轨迹为成功轨迹的比例。

结论：UI-Evol 对轨迹质量具有鲁棒性，即使输入较差轨迹也能有效进化知识。

### 知识迁移实验（Table 3）

用 o3 生成的轨迹进化知识后，提供给 GPT-4o 使用：

| 知识来源 | GPT-4o 成功率 |
|---------|-------------|
| 原始 Web 知识 | 19.5% |
| 从 GPT-4o 轨迹进化 | 22.0% |
| 从 o3 轨迹进化 | 22.4% |

进化知识**可跨模型迁移**，o3 轨迹进化的知识甚至略优于 GPT-4o 自身轨迹。

### Case Study

任务："将 LibreOffice Writer 文件中每个单词首字母大写"。

- Web 知识建议"鼠标拖拽选中全文"→ agent 只选中了部分文本 → 任务失败
- UI-Evol Retrace 阶段准确识别出 agent 仅选中了部分段落
- Critique 阶段诊断为"输出/屏幕误解"，推荐使用 **Ctrl+A** 代替鼠标拖拽
- 进化后知识修正了步骤 2，后续执行成功

## 计算成本

| 阶段 | 时间 | Token 消耗/任务 | 模型 |
|-----|------|----------------|------|
| 原始轨迹收集 | ~2h (30 实例并行) | — | GPT-4o |
| Retrace | ~1h (12 线程) | ~85k input + 400 output | GPT-4o |
| Critique | 含在上述 1h | ~800 input + 150 output | OpenAI-o3 |
| **总计** | ~3h | 平均 $0.22/任务 | 全基准 ~$81 |

## 局限性

1. **依赖强基线系统**：UI-Evol 是 Agent S2 的插件，未验证在其他 agent 框架上的泛化性（虽声称 plug-and-play）
2. **仅在 OSWorld 上验证**：369 个任务的规模有限，未在 WebArena、AndroidWorld 等更多 benchmark 上测试
3. **单轮进化**：当前仅做一轮 "执行→进化"，未探索多轮迭代进化的效果和收敛性
4. **Retrace 依赖 LMM 视觉能力**：若截图差异微妙（如文本内容微调），LMM 可能遗漏关键变化
5. **成本非零**：每个任务 $0.22 的 API 成本在大规模部署时需考量
6. **未讨论知识冲突**：多次进化后知识库条目可能出现矛盾，缺乏冲突消解机制

## 可复现性要点

- 基准代码：OSWorld（开源）
- Agent 框架：Agent S2（开源）
- 搜索引擎：Perplexica（开源，但需要自建实例）
- 骨干模型：GPT-4o、OpenAI-o3（闭源，需 API）
- 知识快照冻结、温度固定为 0
- 并行评测框架基于 Azure ML Jobs（可复现但需云资源）
- 每个实验重复 3 次以报告标准差

## 个人点评

**优点**：
- 问题定义精准：90% 知识正确 → 41% 成功率，数据很有说服力地揭示了 knowledge-execution gap
- 方法设计务实：Retrace 用视觉证据代替主观报告是一个简洁但关键的设计选择；Critique 的四步 CoT 推理结构清晰
- 实验严谨：首次在 GUI agent 研究中系统关注 **行为方差**，并通过重复实验量化，这比只报告单次最佳结果诚实得多
- 知识可迁移性实验有实用价值

**不足**：
- 改进幅度相对有限（19.5→22.0），绝对性能仍有很大提升空间
- 仅在 Agent S2 + OSWorld 上验证，通用性需要更广泛的实验支撑
- 没有分析哪些类型任务从进化中获益最大，缺乏细粒度分析
- 未与同期 Self-Refine、Reflexion 等自我改进方法做直接对比

**总体**：本文切入了一个真实且被低估的问题（知识可用性 vs 知识正确性），方法朴素但有效，标准差的降低比绝对性能提升更有意义——这使得 agent 的行为更可预测、更适合部署。作为 ICML 论文，其贡献在于问题发现和实验方法论（并行评测+重复实验报告方差）而非方法本身的复杂性。
