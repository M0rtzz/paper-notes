# Towards Benchmarking Foundation Models for Tabular Data With Text

**会议**: ICML2025
**arXiv**: [2507.07829](https://arxiv.org/abs/2507.07829)
**代码**: 待确认
**领域**: self_supervised
**关键词**: 表格数据, 文本特征, 基础模型, 基准测试, TabPFN

## 一句话总结
首个系统性研究表格基础模型处理文本特征能力的工作：手动策划含语义丰富文本列的真实数据集集合，比较fastText/Skrub/AutoGluon三种文本嵌入策略，揭示现有方法对语义模式的捕捉仍有显著不足。

## 研究背景与动机

### 表格+文本的现实需求
真实表格数据常含自由文本字段（职位描述/临床笔记/产品摘要），但现有基准几乎不包含文本列。

### 当前方法的分歧
- AutoGluon: 文本→稀疏TF-IDF
- CARTE: fastText句向量
- TabPFNv2 API: 接受原始文本但方法不公开

### 核心问题
哪种嵌入策略最好？什么条件下？

## 方法详解

### 文本嵌入策略
1. **fastText**：预训练词向量平均
2. **Skrub TableVectorizer**：n-gram TF-IDF
3. **AutoGluon文本编码器**：AutoML集成

### 定性反例分析
构造简单合成任务显示：
- TF-IDF在未见同义词上失败（n-gram break）
- fastText在噪声下退化（simple NLP break）
- BERT类模型在语义歧义上失败（LLM break）

### 策划的真实基准
手动从OpenML/Kaggle筛选含信息丰富文本列的数据集，覆盖多领域多样本量。

## 实验关键数据

### 合成反例实验

| 方法 | N-Gram Break | NLP Break | LLM Break |
|------|-------------|----------|----------|
| TF-IDF | 失败 | 可行 | 可行 |
| fastText | 可行 | 失败 | 可行 |
| BERT | 可行 | 可行 | 失败 |

### 真实数据集基准(XGBoost+嵌入)

| 嵌入策略 | 平均准确率 | 最佳数据集比例 |
|---------|----------|--------------|
| 无文本(基线) | 低 | 0% |
| fastText | 中 | ~20% |
| Skrub | 中 | ~30% |
| AutoGluon | 中-高 | ~40% |
| TabPFNv2 API | 中-高 | ~35% |

### 关键发现
1. 没有单一嵌入方法在所有数据集上最优
2. 所有方法都存在系统性失败模式
3. 语义丰富的文本列确实对预测有显著帮助
4. 需要更好的面向表格任务的文本嵌入方法

## 亮点与洞察

1. 合成反例精确暴露了每种方法的失败模式——比泛泛对比更有指导性。
2. 手动策划的基准填补了评测空白。
3. "没有赢家"的结论对社区有警示价值。
4. 指明了表格+文本的明确研究方向。

## 局限性 / 可改进方向

1. 策划数据集数量有限（目前约10个）。
2. 未测试最新的instruction-following嵌入模型(如E5/GTE)。
3. 多文本列的联合处理未讨论。
4. 缺乏与row-as-text方法(如TabLLM)的对比。

## 相关工作与启发

- 与TabPFN系列的关系：本文评估TabPFNv2对文本的处理。
- 与Grinsztajn et al.的关系：他们比较30+嵌入方法但不聚焦基础模型。
- 启发：表格基础模型需要原生的文本处理能力而非简单拼接。

## 补充分析

### 三种嵌入策略的适用场景
fastText适合简短文本、Skrub适合粗糙分类型文本、AutoGluon适合多样性大的场景。但没有一种统治所有场景。

### TabPFNv2的文本处理“黑箱”
TabPFNv2 API接受原始文本但不公开处理方法——这使得公平对比困难。本文呼吁表格基础模型应开源文本处理流程。
### 对模型设计的具体建议
表格基础模型需要原生的文本处理能力（而非简单嵌入拼接），理想方案是预训练阶段就见过文本列。
