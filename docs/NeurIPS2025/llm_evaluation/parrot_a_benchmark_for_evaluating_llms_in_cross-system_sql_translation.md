# PARROT: A Benchmark for Evaluating LLMs in Cross-System SQL Translation

**会议**: NeurIPS 2025  
**arXiv**: [2509.23338](https://arxiv.org/abs/2509.23338)  
**代码**: 有 (code4db.github.io/parrot-bench)  
**领域**: LLM/NLP / 数据库  
**关键词**: SQL翻译, 跨系统SQL, 数据库方言, LLM评测, 基准数据集

## 一句话总结

提出 PARROT，首个面向跨数据库系统 SQL 翻译 (SQL-to-SQL) 的实用基准，包含598个核心翻译对、覆盖22个生产级数据库系统，揭示了 LLM 在系统特定 SQL 方言理解上的显著不足（平均准确率低于38.53%）。

## 研究背景与动机

**Text-to-SQL vs SQL-to-SQL**：虽然 LLM 在 Text-to-SQL 任务上取得了显著进展，但实际工程中同样重要的跨系统 SQL 翻译问题却被忽视。SQL-to-SQL 将一个数据库系统（如 MySQL）上的查询适配为另一个系统（如 ClickHouse）的等价查询。

现有 SQL 基准的不足：

**系统覆盖有限**：多数基准仅关注 SQLite 一种系统

**方言覆盖不足**：无法捕捉各系统特有的函数、数据类型和语法规则

**实用性差距**：未反映真实业务中的 SQL 翻译需求

PARROT 旨在填补这一空白，为 SQL-to-SQL 评估提供标准化、实用化的测试平台。

## 方法详解

### 整体框架

PARROT 基准的构建流程：
1. 从开源项目和真实业务中收集 SQL 翻译案例
2. 标注系统特定的语法差异
3. 构建多维度的测试变体
4. 设计自动化评估流程

### 关键设计

**核心数据集 (PARROT-Core)**：
- 598个翻译对
- 来源：38个开源基准 + 真实业务服务
- 专门设计以挑战系统特定 SQL 理解能力
- LLM 平均准确率低于 38.53%

**PARROT-Diverse（扩展语法测试）**：
- 28,003个翻译对
- 覆盖更广泛的语法差异
- 用于全面评估语法翻译能力

**PARROT-Simple（聚焦压力测试）**：
- 5,306个代表性样本
- 精选最具挑战性的翻译模式
- 用于快速评估核心能力

**系统覆盖**：22个生产级数据库系统，包括MySQL、PostgreSQL、Oracle、SQL Server、ClickHouse、Hive、Spark SQL等。

**方言类型覆盖**：
- 自定义函数（字符串处理、日期时间、数学运算等）
- 数据类型差异（整型、浮点、时间戳等）
- 语法规则（JOIN语法、分页语法、窗口函数等）

### 损失函数 / 训练策略

评估方法：
- 执行准确率 (Execution Accuracy)：翻译后的 SQL 在目标系统上执行是否产生正确结果
- 语法正确率 (Syntactic Correctness)：翻译后的 SQL 是否符合目标系统语法
- 方言覆盖率：LLM 正确处理系统特定特性的比例

## 实验关键数据

### 主实验

在 PARROT-Core 上的 LLM 评估（平均准确率）：

| 模型 | 平均准确率 | MySQL→PG | PG→MySQL | MySQL→CK | 方言理解 |
|------|----------|---------|---------|---------|--------|
| GPT-4 | <38.53% | 中等 | 中等 | 低 | 有限 |
| Claude-3 | <38.53% | 中等 | 中等 | 低 | 有限 |
| Llama-3 | 更低 | 低 | 低 | 极低 | 差 |
| CodeLlama | 更低 | 低 | 低 | 极低 | 差 |

方言类别细分难度：

| 方言类型 | 代表性差异 | 翻译难度 | LLM典型错误率 |
|---------|----------|---------|------------|
| 自定义函数 | DATE_FORMAT vs TO_CHAR | 高 | >60% |
| 数据类型 | INT vs INTEGER vs BIGINT | 中 | ~40% |
| 语法规则 | LIMIT vs TOP vs ROWNUM | 中 | ~35% |
| 系统特性 | 分区表、物化视图语法 | 极高 | >70% |

### 消融实验

| 实验设置 | 影响因素 | 结论 |
|---------|---------|------|
| 系统对数量 | 翻译方向 | 某些方向（如MySQL→ClickHouse）显著更难 |
| 提示工程 | Few-shot vs Zero-shot | Few-shot仅带来有限提升 |
| 方言文档 | 提供系统文档 | 文档提示有一定帮助但不足以解决核心问题 |
| 模型规模 | 参数量 | 更大模型略有提升但差距不显著 |

### 关键发现

1. **LLM SQL方言理解严重不足**：即使最强的LLM在跨系统SQL翻译上的准确率也不到40%
2. **系统特定函数是最大障碍**：自定义函数的翻译错误率最高
3. **翻译方向不对称**：某些翻译方向比反向显著更难
4. **与Text-to-SQL的差距**：SQL-to-SQL需要更深层的系统特定知识

## 亮点与洞察

1. **填补重要空白**：首个系统性的跨数据库SQL翻译基准
2. **实用导向**：数据来源包括真实业务场景，确保实用性
3. **系统覆盖广泛**：22个生产级系统远超现有基准
4. **多变体设计**：Core/Diverse/Simple三种变体满足不同评估需求
5. **公开排行榜**：促进持续的社区参与和改进

## 局限性 / 可改进方向

1. 部分翻译对的等价性验证依赖人工，可能存在标注噪声
2. 数据库系统版本持续更新，基准需要定期维护
3. 未覆盖NoSQL数据库（如MongoDB、Redis的查询翻译）
4. 评估仅关注语义等价性，未考虑性能优化
5. 可以增加更多关于存储过程、触发器等高级特性的翻译测试

## 相关工作与启发

- **Spider/WikiSQL**：Text-to-SQL 基准，但仅覆盖 SQLite
- **SPC/SEATS**：数据库性能基准，非翻译任务
- **SQLGlot**：基于规则的 SQL 翻译工具，PARROT 可作为其评估平台
- 启发：LLM在需要精确的领域特定知识的任务上仍有明显短板
