# Music Arena: Live Evaluation for Text-to-Music

**会议**: NeurIPS 2025  
**arXiv**: [2507.20900](https://arxiv.org/abs/2507.20900)  
**代码**: [有](https://github.com/gclef-cmu/music-arena)  
**领域**: AI评估 / 音乐生成  
**关键词**: text-to-music, 人类偏好评估, live evaluation, 排行榜, Bradley-Terry

## 一句话总结

提出 Music Arena，一个面向文本到音乐（TTM）生成模型的在线实时人类偏好评估平台，通过众包式对比实验收集偏好数据，编制排行榜并定期开放数据。

## 研究背景与动机

文本到音乐（TTM）生成近年进步迅速，但面临两大挑战：

**评估缺乏标准化**：现有听力测试协议不一致（界面、对比模型、标注者分布各异），自动指标（如 FID、FAD）与人类偏好相关性不理想
**偏好数据不可持续**：一次性收集的偏好数据集无法随新模型和用户偏好变化而更新，商业平台数据不开放

**Chatbot Arena** 等 live evaluation 平台已在 LLM、TTS、图像生成领域取得成功，但音乐领域有独特挑战：
- **异构输入输出签名**：不同模型对歌词、时长、人声等支持不同
- **时间性消费**：音乐必须实时聆听，与图像即时感知不同
- 版权和内容安全考量更为复杂

## 方法详解

### 整体框架

Music Arena 由三个模块化组件构成（图1）：
- **前端（Frontend）**：基于 Gradio 的 Web 界面，用户提交文本 prompt → 听两段匿名音频 → 投票偏好
- **后端（Backend）**：核心编排器，处理 prompt、并行调度两个模型生成、同步返回结果
- **模型端点（Model Endpoints）**：统一 Docker 容器封装，适配异构 TTM 系统的不同 API 签名

### 关键设计

1. **LLM 驱动的内容审核与路由系统**：

    - 使用 GPT-4o 解析用户自然语言 prompt，提取结构化信息（是否包含歌词/人声、指定时长等）
    - **审核**：拒绝包含版权音乐引用、文化敏感或不当内容的 prompt
    - **路由**：根据提取的结构化信息将 prompt 分配给兼容的模型子集（如不支持歌词的模型不会参与含歌词的对比）

2. **细粒度偏好收集**：

    - 除四选一偏好（A好/B好/平局/都差）外，记录**聆听行为数据**：每段音频的播放/暂停时间戳、总聆听时长
    - 强制最低聆听时长（4秒）后才解锁投票界面
    - 投票后鼓励提供自然语言反馈
    - 隐藏音频时长以避免长度偏见

3. **隐私保护与数据开放策略**：

    - IP 地址通过加盐哈希进行假名化，永远不存储原始标识符
    - 全平台开源（除密钥外）
    - **滚动月度数据发布**，提供可持续的偏好数据源
    - 排行榜额外展示训练数据信息和生成速度（中位 RTF）

### 支持的模型

| 类型 | 模型 | 特点 |
|------|------|------|
| 开源 | MusicGen (Meta) | 仅器乐 |
| 开源 | Stable Audio Open (Stability AI) | 支持指定时长 |
| 开源 | SongGen | 支持歌词（需GPT-4o生成） |
| 开源 | ACE-Step (ACE Studio) | 支持歌词 |
| 开源 | Magenta RealTime (Google) | 仅器乐 |
| 商业 | FUZZ 1.0/1.1 (Producer.ai) | 联合生成歌词+音频 |
| 商业 | Stable Audio 2.0 | 支持指定时长 |
| 商业 | Lyria RealTime (Google) | 仅器乐 |

论文在附录中提供了初步 Bradley-Terry 排行榜，基于收集到的 1,051 次投票编制。具体模型排名和 Arena Score 作为平台上线初期的验证性结果呈现。

| 关键观察 | 说明 |
|----------|------|
| 商业模型普遍评分较高 | FUZZ 等商业系统在偏好对比中表现靠前 |
| 开源与商业差距存在但在缩小 | ACE-Step 等开源歌词模型有竞争力 |
| 用户更偏好有声乐的输出 | 含歌词/人声的生成更受青睐 |
| 平均每用户参与约3.8场 battle | 用户粘性有待提升 |

### 关键发现

- 音乐领域 TTM 系统的**类型签名异构性**远比其他 AI 领域严重，统一评估框架的设计至关重要
- 聆听行为数据揭示用户在投票前的消费模式，为理解偏好形成机制提供了新维度
- 自然语言反馈可以提供比二元偏好更丰富的洞察

## 亮点与洞察

- **领域适配是关键**：live evaluation 不能简单复制 Chatbot Arena 的做法，音乐的时间性消费、版权敏感性和异构系统签名都需要定制化设计
- **LLM 作为中间层**的路由/审核系统是解决异构系统统一评估的优雅方案
- **滚动数据发布**解决了偏好数据随时间过时的问题
- 排行榜不只看偏好分数，还展示训练数据和速度信息，体现了负责任的评估理念

## 局限性 / 可改进方向

- 用户群体偏向美国用户和 AI 爱好者，代表性不足
- 目前仅支持 text-to-music，不含风格迁移、符号音乐生成等任务
- 模型配对为均匀随机选择，未优化配对策略
- 自托管开源模型的长期可持续性是挑战
- 无法追踪用户在音频内的 seeking 行为
- 初始数据量（1,051票）相对较小，排行榜统计显著性有限

## 相关工作与启发

- Chatbot Arena（LLM）、TTS Arena 2.0、GenAI Arena（图像/视频）——live evaluation 范式的先驱
- MusicRL（Cideron et al.）——用偏好数据对齐 TTM 系统的方向
- 为其他具有独特领域特性的 AI 任务（如 3D 生成、代码辅助）提供 live evaluation 适配的参考
