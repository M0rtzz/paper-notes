# AgentTTS: Large Language Model Agent for Test-time Compute-optimal Scaling Strategy in Complex Tasks

**会议**: NeurIPS 2025  
**arXiv**: [2508.00890](https://arxiv.org/abs/2508.00890)  
**代码**: [GitHub](https://github.com/FairyFali/AgentTTS/)  
**领域**: LLM Agent / 测试时扩展  
**关键词**: 测试时扩展, 计算最优策略, 多阶段任务, LLM Agent, 预算分配

## 一句话总结

研究多阶段复杂任务中的测试时计算最优扩展问题，通过先导实验提炼三个通用洞察，据此设计 AgentTTS——一个基于 LLM agent 的框架，通过迭代反馈驱动交互自主搜索最优模型+预算分配方案。

## 研究背景与动机

- 测试时扩展（TTS）通过推理时分配更多计算来提升 LLM 性能（如重复采样+验证/融合）
- 现有 TTS 研究集中在**单阶段任务**（数学题、代码生成），但现实中许多问题是**多阶段复杂任务**
- 多阶段任务示例：检索+问答、需求分析→设计→编码→测试、任务分解→工具选择→参数预测
- **两大挑战**：
    1. 搜索空间爆炸：3 个子任务 × 2 种模型 → 高达 $10^6$ 种配置
    2. 子任务间相互依赖：上游子任务的预算影响下游的最优分配

## 方法详解

### 整体框架

AgentTTS 包含三个核心组件：

1. **Agent**：LLM agent，生成候选配置（试验）和指导方针
2. **Archive**：存储历史试验、反馈和指导方针
3. **Environment**：在实际任务平台上执行试验并返回性能反馈

迭代搜索流程：Agent 生成试验 → Environment 评估 → Agent 总结指导方针 → 生成新试验 → 循环

### 关键设计

1. **三个经验洞察驱动搜索**:
    - 功能：从 4 种任务类型、6 个数据集的先导实验中提炼通用规律
    - 核心思路：
        - **Insight 1**：不同子任务偏好不同规模的模型（检索偏好大模型，QA 偏好小模型多采样）
        - **Insight 2**：每个子任务存在最优预算点，超出后收益递减甚至为负
        - **Insight 3**：上游子任务的预算分配影响下游的模型选择和最优预算
    - 设计动机：将领域知识注入 LLM agent 的搜索过程，避免盲目探索

2. **初始化阶段的模型偏好估计**:
    - 功能：对每个子任务，在最大预算下对比所有候选模型，确定模型偏好
    - 核心思路：固定其他子任务为最大模型单次推理，逐个子任务比较模型性能
    - 设计动机：基于 Insight 1 快速缩小搜索空间

### 损失函数 / 训练策略

- 统一预算转换：以最小模型在最低消耗任务上的单次推理 FLOPs 为基本单位

$$B = f_{\text{budget}}(M_\ell, S_\ell, T_\ell) = \frac{2\alpha\beta_2 S_\ell}{\beta_1} + 2(\alpha\beta_2 - 1)$$

- 采用**重复采样+融合**策略进行测试时扩展（温度 0.9）
- AgentTTS 本身不涉及模型训练，是搜索框架

## 实验关键数据

### 主实验（表格）

| 方法 | 搜索效率（步数） | 2WikiMultiHopQA 最优性能 | TaskBench 最优性能 |
|------|---------------|----------------------|-----------------|
| Grid Search | ~100+ | 基准 | 基准 |
| Bayesian Optimization | ~30 | 接近最优 | 接近最优 |
| Random Search | ~50 | 较差 | 较差 |
| **AgentTTS** | **~10-15** | **最优** | **最优** |

### 消融实验

- 去除 Insight 1（无模型偏好初始化）：搜索步数增加 2-3 倍
- 去除 Insight 2/3（无指导方针生成）：容易陷入次优配置
- 不同训练集大小：AgentTTS 对样本量鲁棒，5 条示例即可有效搜索

### 关键发现

- 检索子任务偏好大模型（70B > 8B > 3B），QA 子任务在有限预算下小模型更优
- 高质量检索（F1=0.8）可显著降低 QA 的最优预算；低质量检索（F1=0.35）需要更多 QA 预算
- 子任务级 TTS 存在非单调性能趋势——更多计算不总是更好
- AgentTTS 的指导方针提供可解释的决策理由

## 亮点与洞察

- **新问题定义**：首次正式定义多阶段任务中的计算最优 TTS 问题
- 三个洞察从经验数据中总结，具有跨任务通用性
- LLM agent 作为超参数搜索器，兼具效率和可解释性
- 预算归一化框架使不同模型×任务的比较公平统一

## 局限性 / 可改进方向

- 依赖 LLM agent 的规划能力，搜索质量受限于 agent 模型
- 仅考虑重复采样+融合的 TTS 策略，未涵盖树搜索等方法
- 实验中的多阶段任务最多 3 个子阶段，更多阶段的可扩展性未验证
- FLOPs 作为唯一代价指标，未考虑延迟、内存等实际约束

## 相关工作与启发

- 与 MLCopilot、AgentHPO 等 LLM 超参优化工作类似，但专注于 TTS 场景
- 单阶段 TTS 研究（如 Snell et al.）不适用于多阶段的子任务依赖关系
- Insight 3 表明子任务间的"涟漪效应"是多阶段 TTS 的核心难点
