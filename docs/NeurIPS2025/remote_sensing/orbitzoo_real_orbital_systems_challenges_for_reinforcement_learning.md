# OrbitZoo: Real Orbital Systems Challenges for Reinforcement Learning

**会议**: NeurIPS 2025  
**arXiv**: [2504.04160](https://arxiv.org/abs/2504.04160)  
**代码**: 有（开源）  
**领域**: 强化学习 / 航天动力学  
**关键词**: 多智能体RL, 轨道力学, Orekit, PettingZoo, 碰撞规避

## 一句话总结

提出OrbitZoo——一个基于工业级轨道动力学库Orekit的多智能体强化学习环境，支持碰撞规避、编队飞行等场景，在Starlink真实数据验证中MAPE仅为0.16%。

## 研究背景与动机

随着卫星和轨道碎片数量激增，太空拥堵成为关键问题。碰撞规避、站位保持和轨道机动需要先进的自主决策技术。强化学习在此领域展现出潜力，但现有RL框架存在问题：

- 大多使用从头构建的自定义环境和简化动力学模型（如CR3BP）
- 缺乏标准化，每个工作需独立验证轨道动力学的正确性
- 很少支持多智能体交互、连续控制和高保真扰动力
- 缺少与真实数据的验证，sim-to-real差距大

现有14个RL环境（Table 1对比）中，没有一个同时具备所有关键能力：多智能体、工业级仿真器、高保真动力学、连续控制、真实推力建模、交互可视化和公开代码。

## 方法详解

### 整体框架

OrbitZoo的三大核心模块：

1. **数据生成**：基于Orekit提供高保真轨道数据生成，支持并行传播
2. **强化学习**：标准化为PettingZoo接口，支持MA-POMDP
3. **可定制框架**：模块化设计，支持自定义场景和交互3D可视化

### 关键设计

1. **高保真轨道动力学集成**:
    - 功能：集成Orekit的数值传播器，包含多种扰动力
    - 核心思路：支持高阶重力场、大气阻力、太阳辐射压、第三体引力等扰动；支持Cartesian、Keplerian和equinoctial三种状态表示；支持固定步长和自适应步长（Dormand-Prince）积分
    - 设计动机：简化传播器（如SGP4）精度不足，而Orekit提供了高保真且经过验证的数值传播

2. **多智能体RL标准化接口**:
    - 功能：通过PettingZoo库实现MA-POMDP标准接口
    - 核心思路：每个卫星作为独立智能体，具有部分可观测性；支持合作、竞争和混合场景；推力模型使用极坐标参数化（推力大小、偏转角、方位角），可转换为Cartesian推力向量
    - 设计动机：现有方法多为单智能体全观测设定，不符合真实场景中通信受限和分散决策的需求

3. **真实星座验证**:
    - 功能：使用Starlink卫星星座数据验证仿真精度
    - 核心思路：通过调整阻力系数和反射系数等物理参数，将数值传播与星历数据对齐
    - 设计动机：弥合sim-to-real差距，确保RL策略在真实环境中的可转移性

### 损失函数 / 训练策略

OrbitZoo支持多种RL算法和奖励设计：

- 支持PPO、SAC、DQN、DDPG、MAPPO等算法
- 灵活的物理接地奖励函数（如燃料消耗、距离保持、碰撞概率等）
- 支持多目标RL（MORL），可平衡燃料-时间-风险等权衡
- 兼容EPyMARL和MARLlib等MARL库

### 主实验（表格）

### 消融实验

- 不同积分方法对精度的影响：自适应Dormand-Prince优于固定步长RK4
- 不同状态表示（Keplerian vs Equinoctial vs Cartesian）对RL训练的影响：Equinoctial避免了圆轨道和赤道轨道的奇点
- 并行传播的扩展性测试

### 关键发现

- OrbitZoo是首个同时满足所有7项关键能力的轨道RL环境
- 0.16% MAPE表明仿真可靠地反映真实轨道行为
- 高保真动力学对RL策略的可转移性至关重要——简化模型训练的策略在真实条件下可能失效
- PettingZoo集成使得标准MARL算法可直接使用

## 亮点与洞察

- **完整性**：唯一同时具备所有关键能力的轨道RL环境
- **实用性**：0.16% MAPE的真实验证确保sim-to-real可信度
- **模块化**：清晰的抽象层分离使用户可轻松扩展
- 系统梳理了将RL应用于轨道动力学的核心挑战（Table 2非常有价值）

## 局限性 / 可改进方向

- 缺少使用OrbitZoo训练RL智能体的端到端实验结果（仅提供环境，未展示策略训练）
- 高保真数值传播的计算开销限制了大规模星座（数千卫星）的实时训练
- 未讨论通信延迟和带宽约束对分布式MARL的影响
- 缺少与简化环境（如CR3BP-based）训练的策略迁移对比
- 3D可视化组件的性能和可扩展性未详细评估

## 相关工作与启发

- **Orekit**：Java开源轨道力学库，本文将其与Python RL生态系统桥接
- **PettingZoo**：标准化MARL接口，使OrbitZoo兼容大量已有算法
- **ColAvGym (Kazemi 2024)**：单智能体碰撞规避环境，使用真实CDM数据但不支持多智能体
- 为航天RL社区提供了标准化基准，类似于Gym/MuJoCo之于机器人RL的地位

## 评分

⭐⭐⭐ — 工程贡献扎实，功能最完整的轨道RL环境，但缺少端到端RL训练实验验证
