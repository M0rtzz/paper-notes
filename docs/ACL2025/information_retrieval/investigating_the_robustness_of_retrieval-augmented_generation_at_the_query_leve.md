# Investigating the Robustness of Retrieval-Augmented Generation at the Query Level

**会议**: ACL 2025  
**arXiv**: [2507.06956](https://arxiv.org/abs/2507.06956)  
**代码**: 有（将公开）  
**领域**: NLP / 检索增强生成鲁棒性  
**关键词**: RAG鲁棒性, 查询扰动, 检索器敏感性, 生成器敏感性, 评估框架

## 一句话总结

系统研究 RAG 管道对查询级扰动（冗余插入、正式语气、歧义引入、拼写错误）的敏感性，提出解耦各模块鲁棒性的分析框架，发现检索器是性能瓶颈且不同扰动类型的主导因素不同。

## 研究背景与动机

RAG 系统通过在推理时动态结合外部知识来增强 LLM 回答的准确性，减少幻觉。然而，RAG 系统的一个关键实际问题是**对输入查询质量的强依赖**：

1. **用户表述差异**：不同用户提出同一问题的措辞可能截然不同
2. **拼写错误普遍**：尤其在快速输入场景下
3. **稳健性关乎公平**：查询表述差异可能反映用户的教育和文化背景差异

已有研究要么聚焦检索器鲁棒性，要么聚焦上下文噪声对生成器的影响，**缺乏对 RAG 管道各模块在查询扰动下的系统性解耦分析**。本文首次提供了一个框架来解耦 RAG 管道中各模块对查询扰动的敏感性。

## 方法详解

### 整体框架

作者设计了一套查询扰动生成→隔离评估→端到端分析→相关性溯因的完整分析流程：
1. 生成多类型查询扰动
2. 分别评估检索器和生成器的鲁棒性
3. 评估端到端 RAG 性能
4. 通过 Pearson 相关性定位敏感模块
5. 分析 LLM 内部表示变化

### 关键设计

#### 1. 查询扰动策略（5种类型）

**基于 LLM 提示的扰动**（3种）：

- **冗余插入（Redundancy）**: 向查询添加不增加信息价值的内容，如 "I'm curious to know the specific dates..." 替代 "when does..."
- **正式语气变换（Formal Tone）**: 将查询改为更正式的表述，语义不变
- **歧义引入（Ambiguity）**: 插入 "might" 等模糊词或用更泛化的词替换（如 "rapper"→"artist"）

**基于拼写错误的扰动**（2种）：

- **10% 拼写错误**: 对 10% 的词进行基于 QWERTY 键盘邻近键的字符替换
- **25% 拼写错误**: 对 25% 的词进行同类替换

每种扰动对每个原始查询生成5个变体。使用 TextAttack 库模拟拼写错误，保留停用词不被修改。

#### 2. 隔离评估机制

- **检索器评估**：用 Recall@k 衡量扰动前后检索性能变化
- **生成器评估**：设计 closed-book（纯参数知识）和 oracle（完美检索）两种设置，用 Match 指标评估
- **端到端评估**：组合检索器和生成器，使用 Match 指标

#### 3. 相关性分析

- 计算检索器-RAG 和生成器-RAG 之间的 Pearson 相关系数
- 根据不同扰动类型的相关性差异，定位 RAG 管道中的敏感瓶颈

#### 4. 内部表示分析

- 提取 LLM 最后隐层最后一个 token 的注意力头平均表示
- 用 PCA 降维可视化不同扰动下表示的分散程度

### 实验设置

- **数据集**：NQ（通用QA, 1496样本）、HotpotQA（多跳QA, 1494样本）、BioASQ（生物医学, 378样本）
- **检索器**：BGE-base-en-v1.5、Contriever（稠密），BM25、BM25-Multi（稀疏）
- **生成器**：Llama-3.1-8B-Instruct、Mistral-7B-Instruct-v0.2、Qwen2.5-7B-Instruct
- **共 12 个 RAG 管道组合**，总计 1092+ 实验

## 实验关键数据

### 检索器鲁棒性（核心发现）

| 检索器类型 | 冗余插入 | 正式语气 | 歧义引入 | 拼写10% | 拼写25% |
|-----------|---------|---------|---------|--------|--------|
| 稠密检索器 | 较鲁棒 | 影响最小 | 中等下降 | 明显下降 | 大幅下降 |
| 稀疏检索器 | **大幅下降** | 影响最小 | 中等下降 | 较鲁棒 | 中等下降 |

- 稠密检索器对冗余信息更鲁棒，但对拼写错误敏感
- 稀疏检索器对拼写错误更鲁棒，但对冗余信息极度敏感（特别是 BioASQ 领域数据集）
- 正式语气变换对所有检索器影响最小

### 生成器鲁棒性

| 关键发现 | 详情 |
|---------|------|
| 歧义最具挑战 | 在 closed-book 和 oracle 两种设置中均导致性能下降 |
| 拼写错误影响 | 在 closed-book 下影响大，oracle 下（10%）可恢复 |
| 冗余对领域数据敏感 | 对 NQ/HotpotQA 影响小，对 BioASQ 影响大 |
| 上下文缓解作用 | 提供正确文档可大幅缓解查询扰动的负面影响 |

### Pearson 相关性分析（表2）

| 扰动类型 | BioASQ 检索器-RAG | BioASQ Oracle-RAG | NQ 检索器-RAG | NQ Oracle-RAG |
|---------|------------------|------------------|-------------|-------------|
| 冗余 | 0.05 | 0.35↑ | 0.31↑ | 0.11 |
| 正式语气 | 0.04 | 0.15↑ | 0.27↑ | 0.14 |
| 歧义 | 0.15 | 0.33↑ | 0.30↑ | 0.15 |
| 拼写10% | 0.21↑ | 0.04 | 0.35↑ | 0.06 |
| 拼写25% | 0.23↑ | 0.12 | 0.40↑ | 0.03 |

- NQ 数据集：端到端性能主要由检索器决定
- BioASQ 数据集：拼写错误由检索器主导，歧义/冗余由生成器主导
- 不同扰动类型的敏感瓶颈不同

### 关键发现

1. **RAG 性能趋势主要由检索器决定**，但在领域特定数据集上生成器的作用不容忽视
2. **拼写错误（25%）是最致命的扰动**——在所有组合中均导致最大性能下降
3. **closed-book 性能不能预测 oracle 性能**——强调了上下文利用能力的重要性
4. 扰动导致 **LLM 内部表示分散**——冗余和歧义引起的分散最为显著
5. **无需复杂方案**——理解自身管道中哪个模块对何种扰动敏感，即可有针对性地加固

## 亮点与洞察

- **解耦分析框架**的设计理念优秀——不是泛泛测整体鲁棒性，而是精确定位每类扰动的敏感模块
- **closed-book vs oracle** 对比揭示了关键洞见：即使 LLM 自身不鲁棒，好的检索（或完美检索）也能补偿
- **实验规模大**（1092+ 实验），覆盖面广（3数据集×4检索器×3生成器×5扰动类型）
- **实践导向强**：直接给出针对性建议，帮助开发者诊断和加固自己的 RAG 系统

## 局限性 / 可改进方向

- 未探索缓解策略（如查询改写、对抗训练）
- 上下文窗口和输出长度受限，超参搜索不充分
- 未包含重排序器（Reranker）的分析
- 未使用 MRR 和 nDCG 等排序指标，对检索质量的分析不够深入
- 仅使用 Match 指标评估生成质量，可能遗漏流畅性等维度
- LLM 内部表示分析较初步，值得深入挖掘

## 相关工作与启发

- BEIR (Thakur et al., 2021) 提供标准检索基准
- BERGEN (Rau et al., 2024) 提供 RAG 框架和评估工具
- Zhuang & Zuccon (2022) 研究 BERT 检索器对拼写错误的鲁棒性
- 本文启发：可将该框架扩展到多模态 RAG、对话式 RAG、多语言 RAG 等场景
