# SafeRAG: Benchmarking Security in Retrieval-Augmented Generation of Large Language Model

**会议**: ACL 2025  
**arXiv**: [2501.18636](https://arxiv.org/abs/2501.18636)  
**代码**: [https://github.com/IAAR-Shanghai/SafeRAG](https://github.com/IAAR-Shanghai/SafeRAG)  
**领域**: LLM/NLP  
**关键词**: RAG安全, 数据注入攻击, 检索增强生成, 安全评估基准, LLM安全

## 一句话总结

提出 SafeRAG 安全基准，定义了四种增强型攻击任务（银噪声、上下文间冲突、软广告、白色拒绝服务），通过在 RAG 管道不同阶段注入攻击文本来系统评估 14 种 RAG 组件的安全漏洞，揭示了即使最明显的攻击也能轻松绕过现有防御。

## 研究背景与动机

检索增强生成（RAG）通过整合外部知识来扩展 LLM 的知识边界，已广泛应用于 ChatGPT、Gemini、Perplexity.ai 等产品。然而，RAG 管道中查询相关文本依次经过检索器、过滤器和生成器的处理流程引入了潜在安全风险——攻击者可在任一阶段操纵文本。

现有 RAG 攻击方法存在四个关键短板，无法作为有效的安全评估手段：

**噪声攻击太弱（R-1）**：现有噪声集中在表面相关但实际无关的上下文中，简单的安全过滤器即可防御。

**冲突攻击范围窄（R-2）**：现有冲突主要关注 LLM 已有参数知识与外部文档之间的冲突（context-memory conflict），自适应检索器已能有效缓解。

**毒性攻击过于显式（R-3）**：高级生成器对显式和隐式的偏见、歧视、讽刺等已有较强的检测和规避能力。

**DoS 攻击工程质量低（R-4）**：传统 DoS 通过插入显式或隐式的拒绝信号，但这些信号往往被过滤器拦截或被生成器忽略。

## 方法详解

### 整体框架

SafeRAG 包含三个核心组件：
1. **威胁框架**：定义了四种增强型攻击任务及其在 RAG 管道不同阶段（知识库、已检索上下文、已过滤上下文）的注入方式。
2. **安全评估数据集**：以人工为主、LLM 辅助的方式构建的攻击数据集，覆盖 100 个问答对。
3. **评估指标体系**：包含检索安全指标（RA）和生成安全指标（F1 变体 + ASR）。

### 关键设计

1. **银噪声（Silver Noise）**：不同于传统的不相关噪声，银噪声包含**部分但不完整**的答案。构建过程：先将黄金上下文分解为细粒度命题（最小语义单元），选择与问题语义最相似的命题作为攻击目标，再让 DeepSeek 基于该命题生成 10 个多样化的上下文。这种噪声能绕过大多数安全过滤器（因为确实包含部分有用信息），从而削弱 RAG 的多样性。

2. **上下文间冲突（Inter-Context Conflict）**：不同于 context-memory 冲突，这种攻击让**外部检索到的文档之间互相矛盾**。由于 LLM 缺乏足够的参数知识来处理外部冲突，更容易被操纵过的文本误导。构建规则严格：

    - 最小扰动原则：用最小的修改引入冲突
    - 重写以增加真实感
    - 保留关键事实：避免修改导致冲突失效（如改了年份可能使上下文变得无关而非冲突）

3. **软广告（Soft Ad）**：一种特殊的隐式毒性攻击。将看似专业且无害的软广告无缝嵌入到权威上下文中。两种插入方式：

    - 直接插入：将软广告文本直接加入原始上下文
    - 间接插入：修改原始上下文，将广告与权威实体（政府、组织等）自然关联，使其更隐蔽

4. **白色拒绝服务（White DoS）**：利用安全警告的伪装来绕过过滤器和生成器。攻击文本由三部分组成：

    - 包含原始问题（绕过检索器，确保被检索到）
    - 伪造的安全背景信息（绕过过滤器，看起来有帮助）
    - 虚假指控证据包含大量扭曲事实的警告（绕过生成器，触发拒绝回答）
   
   由于安全警告被视为善意且高优先级的信息，过滤器更不可能拦截，生成器更倾向于采纳。

### 损失函数 / 训练策略

SafeRAG 本身是评估基准而非训练方法，不涉及训练。其数据集构建以人工为主：
- 基础数据来自新华网 2024-08-16 至 2024-09-28 的新闻，覆盖政治、财经、科技、文化、军事五大板块。
- 筛选标准：包含 8+ 连续句子、围绕特定主题、可生成综合性问题（what/why/how）。
- 最终获得 100 个独立的问答对作为基础数据集。
- 冲突和软广告攻击文本由标注员人工构建，噪声和 DoS 由规则+LLM 辅助生成。

## 实验关键数据

### 主实验

评估了 14 种 RAG 组件（4 种检索器、3 种过滤器、8 种生成器）：

| 攻击任务 | 攻击效果排序（按阶段） | 核心发现 |
|---------|----------------------|---------|
| 银噪声 | filtered ctx > retrieved ctx > KB | F1(avg) 随噪声比例增大持续下降 |
| 上下文间冲突 | filtered ctx > retrieved ctx > KB | RAG 难以判断哪条信息为真 |
| 软广告 | filtered ctx > retrieved ctx > KB | RAG 误将伪装的权威陈述当事实 |
| 白色DoS | filtered ctx > retrieved ctx > KB | 即使检索到证据仍拒绝回答 |

检索器鲁棒性排序（噪声）：**Hybrid-Rerank > Hybrid > BM25 > DPR**

### 消融实验

| 配置 | 关键指标 | 说明 |
|------|---------|------|
| 不同检索器 vs 不同攻击 | 脆弱性各异 | Hybrid-Rerank 更易受冲突攻击，DPR 更易受 DoS |
| SKR 压缩器 vs NLI 过滤器 | 互有优劣 | SKR 在毒性/DoS 中有效压缩攻击内容，但在噪声/冲突中反而压缩了有用信息 |
| NLI 过滤器在毒性/DoS | AFR ≈ 无过滤 | NLI 过滤器对软广告和白色DoS 基本失效 |
| 轻量模型 vs 强模型 | Baichuan 13B 最安全 | GPT 系列等更强模型反而更容易被攻击 |

### 关键发现

- **所有攻击任务都能有效绕过现有 RAG 组件**：即使是最明显的攻击也能导致 RAG 服务质量下降。
- **攻击阶段越靠后越危险**：注入到已过滤上下文比注入到知识库的攻击效果更强（因为不需经过额外的防御层）。
- **更强的模型不一定更安全**：GPT-4o 等模型反而可能对本文提出的新型攻击（特别是毒性和 DoS）更敏感，因为它们对"安全警告"的响应更积极。
- **评估指标与人类判断高度一致**：ASR/AFR 和 F1 指标使用 DeepSeek 作为评估器时人类一致性达到 89-100%。
- **首个中文 RAG 安全基准**：填补了中文场景下 RAG 安全评估的空白。

## 亮点与洞察

- **攻击设计巧妙**：四种攻击都针对现有防御的具体弱点设计，银噪声利用"部分正确"绕过过滤器，白色 DoS 利用"安全警告"的权威性绕过生成器，非常贴近真实攻击场景。
- **威胁模型系统全面**：同时考虑攻击在知识库、检索上下文、过滤上下文三个阶段的注入，形成了完整的攻击面分析。
- **人工为主的数据构建**：特别是冲突上下文的构建，通过严格的标注规则避免了 LLM 生成冲突时常见的"无效冲突"问题。
- **反直觉发现**：更强的 LLM 反而更不安全——这揭示了 LLM"乐于助人"（helpfulness）和安全性之间的深层矛盾。
- **评估框架可复用**：F1 变体和 ASR 指标设计合理，可推广到其他 RAG 安全评估场景。

## 局限性 / 可改进方向

1. 仅覆盖数据注入攻击，未评估模型后门攻击等其他安全威胁。
2. 仅针对单模态文本 RAG，未考虑多模态 RAG（图像、结构化知识图谱、音频）的安全性。
3. 评估基准规模较小（100 个问答对），可能不足以覆盖所有攻击变体。
4. 仅在中文上评估，跨语言泛化性未验证。
5. 未提出具体的防御方法，仅揭示漏洞——未来需开发针对性防御策略。

## 相关工作与启发

- RGB (Chen et al., 2024a) 和 RAG Bench (Fang et al., 2024) 是现有的 RAG 噪声和冲突评估工作，但攻击方法过于简单。
- PoisonedRAG (Zou et al., 2024) 关注知识毒化攻击，但主要通过 LLM 生成反事实扰动，质量不如人工构建。
- Phantom (Chaudhari et al., 2024) 和 MAR (Shafran et al., 2024) 研究 DoS 攻击，但攻击文本容易被过滤或忽略。
- 该工作对 RAG 产品化具有重要警示意义：当前的 RAG 系统在面对精心设计的攻击时极其脆弱，亟需开发更鲁棒的检索、过滤和生成组件。
