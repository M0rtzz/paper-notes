# DARS: Dynamic Action Re-Sampling to Enhance Coding Agent Performance by Adaptive Tree Traversal

**会议**: ACL 2025  
**arXiv**: [2503.14269](https://arxiv.org/abs/2503.14269)  
**代码**: 无  
**领域**: LLM Agent  
**关键词**: 编程智能体、推理时计算扩展、树搜索、动态重采样、SWE-Bench

## 一句话总结

本文提出 DARS（动态动作重采样），一种针对编程智能体的推理时计算扩展方法，在智能体做出次优决策的关键节点上动态分支并尝试替代动作，在 SWE-Bench Lite 上以 Claude 3.5 Sonnet V2 实现 55% 的 pass@k 和 47% 的 pass@1，超越当时开源 SOTA 框架。

## 研究背景与动机

**领域现状**：LLM 驱动的编程智能体（如 SWE-Agent、Aider、OpenDevin）在自动化软件工程任务中展现了巨大潜力，能够自动定位 bug、编写补丁、完成复杂开发任务。常见的推理模式是线性轨迹执行——智能体按步骤依次生成动作并执行。

**现有痛点**：线性轨迹方法的核心问题是一旦智能体在某个步骤做出了错误决策，后续所有操作都建立在这个错误基础上，很难恢复。现有的计算扩展策略（如简单的 best-of-N 随机采样）虽然可以通过并行生成多条轨迹来提高成功率，但效率低下，因为多条轨迹可能在不同位置犯同样的错误，浪费大量计算资源。

**核心矛盾**：线性执行缺乏容错性 vs. 全随机采样效率低下。需要一种"有针对性"的回溯机制，在检测到错误时精准地分支出替代路径，而不是从头开始。

**本文目标**：设计一种高效的推理时计算扩展策略，能够利用执行反馈识别关键决策点，并在这些点上智能地分支出替代动作路径。

**切入角度**：作者观察到编程任务有天然的执行反馈（如测试通过/失败、编译错误等），可以作为判断决策质量的信号。通过利用这些反馈，可以精准定位需要回溯的节点。

**核心 idea**：在编程智能体的执行轨迹中，当检测到次优决策时，不是重新开始，而是回到该决策点，利用之前轨迹的历史和执行反馈信息来生成一个替代动作，形成一棵自适应的决策树。

## 方法详解

### 整体框架

DARS 的执行流程是：(1) 智能体开始线性执行任务并记录每步动作和反馈；(2) 当某步执行失败或触发预设的分支条件时，将该步标记为"关键决策点"；(3) 回退到该决策点，将之前的轨迹历史和这次尝试的失败反馈作为上下文，提示 LLM 生成一个替代动作；(4) 从替代动作继续执行；(5) 如果替代路径也失败，继续回退到更早的决策点或尝试其他替代动作，形成自适应树遍历。

### 关键设计

1. **关键决策点识别**:

    - 功能：确定轨迹中哪些步骤是值得回溯和重试的
    - 核心思路：利用执行反馈（如测试结果、错误信息、编译状态）来判断某个动作是否导致了"次优"结果。当检测到执行失败或代码编辑引入新错误时，该步骤被标记为关键决策点。这比随机选择分支点要精准得多
    - 设计动机：不是所有步骤都值得回溯——只有那些导致后续失败的"关键拐点"才需要被修正。这样可以大幅减少搜索空间

2. **反馈感知的替代动作生成**:

    - 功能：在关键决策点生成知情的替代动作
    - 核心思路：将完整的轨迹历史（到该决策点为止的所有动作和观察）以及之前尝试的动作和其失败反馈一并作为提示输入给 LLM。这样 LLM 不仅知道"该做什么"，还知道"什么是行不通的"，从而生成真正不同的替代方案
    - 设计动机：普通的随机采样只是在同一上下文下重新生成，很可能产生相似的动作。通过显式提供"失败经验"，可以引导 LLM 探索本质不同的解决路径

3. **自适应树遍历策略**:

    - 功能：控制搜索的广度和深度，在计算预算内最大化成功率
    - 核心思路：采用深度优先搜索策略，优先沿当前分支深入执行，只有当某条路径彻底失败时才回退到上一个分支点。每个决策点允许的替代动作数量有上限，避免无限扩展。整体计算预算通过限制总的动作步数来控制
    - 设计动机：相比广度优先搜索（同时展开多条路径），深度优先策略更适合编程任务——完成一个完整的修复尝试后再决定是否回溯，而不是在多个半成品路径间切换

### 损失函数 / 训练策略

DARS 是一种纯推理时方法，不需要额外训练。它直接利用现有的预训练 LLM（如 Claude 3.5 Sonnet V2）作为策略模型，仅通过精心设计的提示模板和执行反馈来引导决策。

## 实验关键数据

### 主实验

在 SWE-Bench Lite 基准测试上的结果：

| 方法 | pass@1 | pass@k | 基础模型 |
|------|--------|--------|----------|
| SWE-Agent | 18.0% | — | GPT-4 |
| Aider | 26.3% | — | Claude 3 Opus |
| AutoCodeRover | 30.7% | — | GPT-4o |
| OpenDevin (best-of-N) | 38.0% | 45.0% | Claude 3.5 |
| **DARS** | **47.0%** | **55.0%** | Claude 3.5 Sonnet V2 |

### 消融实验

| 配置 | pass@1 | 说明 |
|------|--------|------|
| 线性轨迹（无分支） | ~38% | 标准单次执行 |
| 随机重采样（best-of-N） | ~42% | 多次独立运行取最佳 |
| DARS（无反馈） | ~44% | 分支但不利用失败反馈 |
| **DARS（完整）** | **47%** | 利用反馈的自适应分支 |

### 关键发现

- **反馈感知的分支是关键**：相比盲目重采样，利用执行反馈信息来指导替代动作的生成带来了最大的收益（约 5 个百分点提升）
- **计算效率优于 best-of-N**：DARS 在使用相同计算预算时，pass@k 比简单的 best-of-N 策略高出约 10 个百分点，因为它避免了在已探索过的失败路径上重复浪费计算
- **对 Claude 3.5 Sonnet V2 效果最佳**：强大的基础模型能更好地利用反馈信息来生成有意义的替代方案

## 亮点与洞察

- **推理时扩展的新范式**：不同于传统的采样-验证范式（如 best-of-N），DARS 提出了一种在执行过程中动态分支的方法，更像是人类程序员的调试思维——"这条路走不通，换个方向试试"
- **反馈利用的精巧设计**：将失败轨迹的信息作为"负面样本"反馈给 LLM，这一思路可以迁移到任何有明确反馈信号的智能体任务（如数学推理、机器人规划等）
- **实用性强**：无需训练即可即插即用于现有编程智能体框架，只需修改推理策略

## 局限与展望

- **依赖强大的基础模型**：DARS 的效果高度依赖 LLM 利用反馈信息的能力，在较弱模型上可能效果有限
- **分支策略的自动化程度不足**：何时分支、分支几次等决策目前依赖启发式规则，理想情况下应该有一个学习到的价值函数来指导
- **只在 SWE-Bench Lite 上评估**：该基准难度适中，未在更难的 SWE-Bench Full 或其他编程任务上验证
- **计算开销随分支数指数增长**：虽然比 best-of-N 更高效，但深层递归分支仍会消耗大量计算资源
- **未与树搜索方法（如 MCTS）对比**：MCTS 等方法可以利用值估计来更智能地选择分支，与 DARS 的结合可能效果更好

## 相关工作与启发

- **vs SWE-Agent**：SWE-Agent 采用线性轨迹执行，无回溯能力。DARS 在其基础上增加了自适应分支，显著提升了成功率
- **vs best-of-N 策略**：多次独立采样缺乏信息共享，不同轨迹可能犯同样的错误。DARS 通过在失败点分支并传递失败信息来避免重复错误
- **vs Tree-of-Thought / MCTS**：ToT 适合推理任务中的搜索，DARS 是类似思路在智能体执行中的应用。区别在于 DARS 利用真实代码执行反馈（而非模型自我评估）来指导搜索

## 评分

- 新颖性: ⭐⭐⭐⭐ 自适应树遍历+反馈感知重采样的组合在编程智能体中是较新的尝试
- 实验充分度: ⭐⭐⭐ 仅在 SWE-Bench Lite 一个基准上评估，消融实验不够详尽
- 写作质量: ⭐⭐⭐⭐ 方法描述清晰，动机论述合理
- 价值: ⭐⭐⭐⭐ 对编程智能体的推理时扩展策略有重要参考价值
