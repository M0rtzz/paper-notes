# Graph-Structured Trajectory Extraction from Travelogues

**会议**: ACL 2025  
**arXiv**: [2410.16633](https://arxiv.org/abs/2410.16633)  
**代码**: 无（计划发布数据集和代码）  
**领域**: 其他  
**关键词**: 轨迹提取, 游记分析, 图结构, 信息抽取, 旅游信息学

## 一句话总结

提出使用图结构（而非序列）表示旅行轨迹，构建了包含地理层级关系（inclusion）和时间顺序关系（transition）的访问顺序图，并建立了日语游记轨迹提取的基准数据集ATD-VSO。

## 研究背景与动机

游记是分析人类旅行行为的重要来源，自动从游记中提取旅行轨迹对旅游服务（如行程规划和推荐）具有重要价值。然而，先前的研究存在两个核心问题：

**轨迹表示不充分**：之前的方法将轨迹简单表示为地点的序列，但当一个地点在地理上包含另一个地点时（如"京都市"和"京都站"），它们无法被简单地线性排列。序列结构无法表达这种层级关系。

**缺乏基准数据集**：之前的研究使用内部数据集，没有公开的带轨迹标注的文本数据集，无法进行公平的比较和研究积累。

作者认为，需要一种能同时表达地理包含关系和时间顺序关系的表示方法，因此提出了访问顺序图（Visiting Order Graph）。

## 方法详解

### 整体框架

任务被分解为两个子任务：
1. **访问状态预测（VSP）**：判断游记中提到的每个地点是否被旅行者实际访问
2. **访问顺序预测（VOP）**：预测已访问地点之间的包含关系和转移关系，构建图结构

### 关键设计

1. **访问顺序图（Visiting Order Graph）**：

    - **节点**：对应被标记为"已访问"的地理实体（geo-entity）
    - **包含关系边（Inclusion）**：蓝色垂直/斜向有向边，表示起始节点在地理上包含终止节点（如"奈良市"包含"东大寺"）
    - **转移关系边（Transition）**：绿色水平有向边，表示旅行者从一个地点直接移动到另一个地点，且两者必须拥有相同的父节点
    - 通过遍历包含和转移关系，可以推断任意实体对之间的访问顺序

2. **访问状态标签体系**：

    - 实体级别：Visit（已访问）、Other（其他）
    - 提及级别：Visit、PlanToVisit（计划访问）、See（看到）、Visit-Past（过去访问）、Visit-Future（将来访问）、UnkOrNotVisit（未知/未访问）
    - 通过提及标签聚合（MLA）规则推导实体标签

3. **特殊情况处理**：

    - Multiple Visits：将重访的实体拆分为子实体
    - UnknownTime：访问时间不明确的实体排除在图之外
    - Overlap：地理上重叠但无包含关系的实体（如"本州"与"东京都"）

4. **序列排序解码（Sequence Sorting Decoding）**：为转移关系预测设计的贪婪解码策略，确保同一父节点下的兄弟节点构成合法的单一序列。

### 损失函数 / 训练策略

- VSP基线：使用LUKE模型（LukeForEntityClassification），两步法——先预测提及标签，再用MLA规则聚合为实体标签
- IRP/TRP基线：使用LUKE模型（LukeForEntityPairClassification），输入两个实体的代表提及所在句子及中间的所有句子，进行成对分类

## 实验关键数据

### 主实验

访问状态预测（VSP）：

| 级别 | 方法 | 准确率 | Macro F1 |
|------|------|--------|----------|
| 提及级 | Majority Label | 0.679 | 0.135 |
| 提及级 | LUKE | 0.789 | 0.468 |
| 实体级 | Majority Label | 0.823 | 0.451 |
| 实体级 | LUKE + MLA | 0.862 | 0.740 |

包含关系预测（IRP）：

| 方法 | 全部 F1 | 深度=1 F1 | 深度≥2 F1 |
|------|---------|-----------|-----------|
| Random | 0.043 | 0.057 | 0.038 |
| Flat | 0.244 | 1.000 | 0.000 |
| LUKE | 0.355 | 0.058 | 0.425 |

转移关系预测（TRP）：

| 方法 | 全部 F1 | 正向 F1 | 反向 F1 |
|------|---------|---------|---------|
| OccOrder (visit status) | 0.758 | 0.794 | 0.386 |
| LUKE (sequence sorting) | 0.748 | 0.796 | 0.366 |
| Random | 0.190 | 0.247 | 0.061 |

### 消融实验

| 配置 | 关键指标 | 说明 |
|------|---------|------|
| LUKE朴素解码 vs 序列排序解码 | 0.680→0.748 | 序列排序解码显著改善TRP |
| 早期提及 vs 访问状态优先 | 0.730→0.758 | 代表提及选择策略影响TRP |
| Visit标签 | F1=0.872~0.918 | VSP最容易的类别 |
| UnkOrNotVisit标签 | F1=0.482~0.561 | VSP最具挑战性的类别 |

### 关键发现

1. 访问状态预测和转移关系预测可以达到较高的准确率，但包含关系预测仍然是巨大挑战（LUKE仅0.355 F1）
2. 基于出现顺序的简单规则（OccOrder）在转移关系预测上就已很强，甚至略优于LUKE基线
3. 包含关系预测困难的根本原因是预训练模型缺乏地理空间知识
4. 标注者一致性较高：包含关系F1=0.94，转移关系F1=0.74

## 亮点与洞察

- **图表示的必要性论证清晰**：通过"奈良市-东大寺-大佛殿"的层级例子直观说明了为何序列不够用
- **任务分解合理**：将复杂的轨迹提取分解为VSP和VOP两个子任务，VOP再分为IRP和TRP
- **Transition要求相同父节点的设计精巧**：通过这个约束，使得不同层级的顺序可以通过遍历推导，避免了边的爆炸式增长
- **数据集构建严谨**：使用了标注者间一致性验证，采用brat和Miro两种工具

## 局限性 / 可改进方向

1. 包含关系预测的F1仅0.355，存在大量提升空间。作者建议融入地理编码特征或利用GeoLM等预训练模型
2. 数据集仅100篇日语游记文档，规模偏小
3. 当前实验基于gold label设置（假设前序任务标签正确），未评估端到端的pipeline性能
4. LUKE基线对ROOT节点的预测很差（F1=0.058），因为目前无法直接预测ROOT
5. 仅支持日语游记，跨语言泛化能力未验证

## 相关工作与启发

- 游记轨迹提取是旅游信息学和人文地理学的交叉领域
- 图结构在实体关系抽取中并不新鲜，但将其应用于轨迹表示是有价值的创新
- GeoLM等融合地理空间信息的预训练模型可能是解决包含关系预测的关键
- 该数据集和任务设定可以启发更多关于空间推理的NLP研究
