# SPOT: Bridging Natural Language and Geospatial Search for Investigative Journalists

**会议**: ACL 2025  
**arXiv**: [2506.13188](https://arxiv.org/abs/2506.13188)  
**代码**: [有](https://github.com/dw-innovation/kid2-spot)  
**领域**: NLP / 地理空间搜索 / 调查新闻  
**关键词**: OpenStreetMap, 自然语言查询, 地理定位验证, LLM微调, YAML生成

## 一句话总结

提出 SPOT——一个面向调查记者的开源自然语言地理空间搜索工具，通过微调 LLaMA 3 将非结构化场景描述转换为 YAML 查询，结合 OSM 标签语义捆绑系统和 Elasticsearch 索引，实现对 OpenStreetMap 的准确自然语言访问。

## 研究背景与动机

调查记者频繁使用 OpenStreetMap (OSM) 进行地理位置验证，但现有工具存在显著门槛：

**技术壁垒高**：Overpass Turbo 等工具要求用户掌握 OverpassQL 查询语言

**现有 AI 工具不足**：GeoGuessr GPT 基于闭源模型且不支持 OSM 集成；GeoSpy 仅支持图像输入；EarthKit 需要手动选择 OSM 标签

**OSM 标签体系复杂**：同一物体可能有多种标签表示方式，且标签组合才能描述一个概念

核心挑战：将非结构化的、可能含有错别字和语法错误的自然语言场景描述，准确转换为 OSM 数据库查询。

SPOT 的独特定位：完全开源、开放权重的 AI 地理位置搜索工具，专门为调查新闻场景设计。

## 方法详解

### 整体框架

SPOT 包含四个主要组件：
1. 标签捆绑构建与索引
2. 合成训练数据生成
3. 模型训练
4. 推理与后处理

### 关键设计

1. **OSM 标签捆绑系统 (Bundle System)**

    - 功能：将自然语言描述器映射到 OSM 标签组合
    - 核心思路：将视觉上相似的 OSM 标签分组，创建静态捆绑列表。例如 "轻轨"、"地铁"、"电车" 都映射到同一个表示"城市小型铁路轨道"的捆绑
    - 索引方式：使用 Elasticsearch 对捆绑进行混合索引——BM25 + SBERT 语义嵌入
    - 设计动机：解决 OSM 标签不一致性问题，且将标签知识与 LLM 解耦（标签更新无需重新训练模型）

2. **合成训练数据生成管道**

    - 先与 OSINT 社区进行用户研究，收集基于图片的场景描述，提取用户需求
    - 设计 YAML 结构：包含搜索区域、实体、属性、空间关系
    - 随机组合 YAML 字段值生成查询组合
    - 使用 7 种人设 + 5 种写作风格作为规格，通过 GPT-4o 将 YAML 转换为自然语言句子
    - 数据集特征：43,976 个训练样本，包含不同长度、多语言地名（含 CJK）、各种错别字和语法错误

3. **模型训练**

    - 基座模型：LLaMA 3
    - 微调方式：LoRA (rank=32, alpha=64)
    - 训练配置：batch size=8, lr=1e-5, weight decay=0.01
    - 早停策略：patience=10, 每 200 步评估

4. **YAML 多层中间表示**

    - 第一层：LLM 输出 YAML 查询（不含 OSM 标签）
    - 第二层：将 YAML 中的实体和属性名称通过 Elasticsearch 替换为最匹配的 OSM 标签捆绑
    - 第三层：转换为实际的 PostGIS 数据库查询
    - 设计动机：将语言理解和领域知识解耦，OSM 标签更新不影响模型

5. **空间关系支持**

    - 精确距离：如"100 米内"
    - 模糊距离："附近"→50 米，"旁边"→50 米（预定义映射表）
    - 包含关系：如"建筑物内的厕所"

### 损失函数 / 训练策略

标准 SFT 训练，使用 unsloth 库进行高效微调。模型通过 HuggingFace Inference Endpoints 部署。

### 数据多样性分析

- UMAP + HDBSCAN 聚类：识别出 **1,274** 个不同语义簇
- **16,416** 个句子未被分配到任何簇（视为噪点/高多样性样本）
- 大量簇和高比例的未聚类样本表明训练数据语义多样性显著

### 关键发现

1. 微调的 LLaMA 3 在 OSM 查询生成任务上显著优于早期的 T5 方案
2. 标签捆绑系统有效应对了 OSM 标签不一致性问题
3. 混合检索（BM25 + SBERT）在处理错别字和同义词方面表现出色
4. 多人设多风格的数据生成策略确保了语义覆盖的多样性

## 亮点与洞察

1. **完全开源方案**：从模型权重到前后端代码全部开源，可自行部署
2. **解耦设计的工程智慧**：LLM 生成的 YAML 不含 OSM 标签，标签匹配由独立的搜索引擎完成——这意味着 OSM 标签体系更新时无需重训模型
3. **面向真实需求**：通过用户研究驱动系统设计，充分考虑了错别字、语法错误、多语言等真实使用场景
4. **实用部署**：已上线运行在 [findthatspot.io](https://www.findthatspot.io/)，集成了 Google Street View 等外部工具
5. **社会价值**：降低了地理位置验证的技术门槛，支持事实核查和反虚假信息

## 局限性 / 可改进方向

1. 依赖 GPT-4o 生成训练数据，存在数据质量受限于教师模型的问题
2. 捆绑列表为静态设计，需要人工维护和更新
3. 当前仅支持文本输入，不支持图像输入（调查记者常需从图像中提取位置线索）
4. 复杂空间关系的推理能力可能有限
5. 对 OSM 数据覆盖度低的区域可能效果不佳

## 相关工作与启发

- OverpassT5 (Staniek et al., 2024)：直接生成 OverpassQL 查询，但要求用户了解 OSM 标签体系
- Lawrence and Riezler (2016)：早期 OSM 自然语言查询方法
- DAIL-SQL、MCS-SQL 等 Text-to-SQL 工作：SPOT 本质上是 Text-to-OSM-Query 问题
