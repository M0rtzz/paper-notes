# EvoWiki: Evaluating LLMs on Evolving Knowledge

**会议**: ACL 2025  
**arXiv**: [2412.13582](https://arxiv.org/abs/2412.13582)  
**代码**: [github](https://github.com/wtangdev/EvoWiki)  
**领域**: LLM/NLP  
**关键词**: 知识演化, 动态基准, RAG, 持续学习, 知识图谱

## 一句话总结

提出 EvoWiki，一个可自动更新的动态评估基准，将知识分为稳定（stable）、演化（evolved）和未知（uncharted）三个层级，用于评估 LLM 在知识持续演化场景下的利用能力，并揭示 RAG 与持续学习（CL）结合具有协同效应。

## 研究背景与动机

现有的 LLM 知识评估基准（如 NaturalQuestion、HotpotQA）大多是静态的，无法跟上 LLM 和知识本身持续演化的步伐。这带来三个关键问题：

**测试集污染**：新发布的 LLM 可能在训练阶段已经接触过静态基准的测试数据，导致评估结果不可靠
**答案过时**：随着知识的更新，静态的标准答案可能已经不再正确，导致评估产生假阴性
**难度差异**：知识是否已在 LLM 训练数据中出现，会显著影响知识利用的难度

虽然已有 LiveBench、Realtime QA 等动态基准尝试解决上述问题，但它们缺乏对知识演化层级的系统性分类，也无法同时评估模型和知识的共同演化。这正是 EvoWiki 要填补的空白。

## 方法详解

### 整体框架

EvoWiki 基于 Wikidata 知识图谱和 Wikipedia 文本构建，具有三大核心特征：

1. **三层级知识分类**：根据 LLM 的知识截止日期，将知识事实分为稳定、演化和未知三类
2. **多维属性**：集成了参考上下文、多跳推理和流行度等多维分析属性
3. **自动可更新性**：构建流程完全自动化，支持基准的持续更新

### 关键设计

**知识演化层级识别**：引入三个关键时间戳来确定知识的演化状态：
- **init-time**（2021年9月）：LLM 训练数据充分覆盖之前的时间点
- **cutoff-time**（2024年1月）：LLM 的知识截止日期
- **current-time**（2024年5月）：评估执行时间

分类规则如下：
- **稳定知识**：从 init-time 到 current-time 保持不变的事实
- **演化知识**：在 init-time 之前已建立，但在 cutoff-time 之后发生变化的事实
- **未知知识**：在 cutoff-time 之后才出现的新事实

**远程监督验证**：将知识三元组锚定到对应的 Wikipedia 页面，确保每个事实可以在文本中找到明确的证据支持，从而过滤噪声数据。

**多跳推理问题生成**：基于精炼后的事实三元组构建多跳推理问题（最多3跳），并利用 GPT-4o-mini 优化问题的自然性。中间跳的三元组被限制为单一宾语，以降低歧义。

**流行度度量**：使用 Wikipedia 页面浏览量作为知识流行度的指标，丰富分析维度。

### 数据统计与质量

最终数据集包含：稳定知识 3,819 个问题、演化知识 3,491 个问题、未知知识 2,954 个问题。人工评估（4位高级研究员对180个样本）显示流畅性、可回答性和准确性均超过 90%。

## 实验关键数据

### 主实验

使用 Llama-3.1-8B-Instruct 和 Mistral-7B-Instruct 两个模型进行评估：

**Closed-book vs Open-book**：
- Closed-book 性能较低（Llama 单跳：31.61%），特别是在演化和未知知识上
- Open-book 提供黄金上下文时性能大幅提升（Llama 单跳：86.87%），证实了外部知识的关键作用

**RAG 性能**：
- Contriever（密集检索）显著优于 BM25（稀疏检索），Llama 上分别达到 77.90% vs 59.41%（稳定/单跳）
- 扩大语料库（从15K到370K文档）导致性能下降，模拟了更真实的检索场景
- RAG 在单跳问题上表现良好，但在多跳问题上效果急剧下降（Contriever: 77.90% → 19.37%）

**CL 性能**：
- SFT 整体优于 CPT，SFT+CPT 的组合在 closed-book 上表现最佳（38.58%/28.84%）
- CL 在所有问题类型上提供了温和但一致的改进

**RAG+CL 协同效应**：
- SFT+Contriever 达到 82.85%/24.02%，显著优于单独使用 RAG 或 CL
- SFT+Open-book 达到 92.10%/60.22%，是所有方法中的最佳组合
- Mistral 上的 SFT+Open-book 甚至在多跳上达到 71.16%

### 消融实验

- 不同 top-k 检索对 RAG 性能的影响：随着 k 增大性能先升后降
- 对演化知识的分析显示，许多模型倾向于给出过时答案而非最新答案（括号中数值为包含过时答案的精确度）

### 关键发现

1. **RAG 在单跳上很强但多跳上严重退化**：多跳推理需要整合多段信息，当前检索方法难以提供完整的推理链
2. **CL 改进温和但一致**：SFT 比 CPT 更有效，可能因为 SFT 直接教模型如何使用知识
3. **RAG+CL 具有协同效应**：结合两种方法的效果优于各自单独使用，暗示混合方法是未来方向
4. **演化知识是最大挑战**：模型经常给出过时答案，说明知识更新机制仍然不足

## 亮点与洞察

- 将知识按时间维度分为三个演化层级是一个非常清晰且实用的框架设计，使评估更加精准
- 自动更新机制使基准不会随时间失效，解决了静态基准的核心痛点
- RAG 与 CL 的协同效应发现具有重要实践意义：单一方法不够，混合策略值得探索
- 流行度属性的加入使研究者可以分析模型对高频 vs 低频知识的处理差异

## 局限性 / 可改进方向

- 知识来源局限于 Wikidata 和 Wikipedia，可能不能代表所有类型的知识演化
- 多跳推理最多3跳，更深层的推理链未被覆盖
- 时间戳设置（init-time=2021.09, cutoff-time=2024.01）是针对特定 LLM 的，不同模型可能需要调整
- 未探索更复杂的知识利用方法（如 knowledge editing、model merging）
- 人工评估样本量相对有限（180个）

## 相关工作与启发

- 与 Realtime QA 相比，EvoWiki 不仅关注新知识，还系统性地考虑稳定和演化知识
- 与 DyKnow 相比，EvoWiki 提供更丰富的属性和多跳推理支持
- 知识冲突研究（Ying et al., 2024b）发现 LLM 倾向于依赖内部知识，即使外部知识正确——这与本文的演化知识问题高度相关
- 可以启发设计更好的知识更新策略：结合检索增强和参数更新的混合方法
