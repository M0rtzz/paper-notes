# Retrieval Models Aren't Tool-Savvy: Benchmarking Tool Retrieval for Large Language Models

**会议**: ACL 2025  
**arXiv**: [2503.01763](https://arxiv.org/abs/2503.01763)  
**代码**: https://github.com/shizhl/Tool-Retrieval-Benchmark (有)  
**领域**: LLM & NLP  
**关键词**: Tool Retrieval, Information Retrieval, Tool Learning, LLM Agent, Benchmark

## 一句话总结

提出 ToolRet——首个大规模工具检索基准（7.6k 任务 + 43k 工具语料库），系统评估了现有 IR 模型在工具检索场景下的表现，发现即使强力检索器也表现不佳，并贡献了超过 200k 训练实例显著提升检索质量。

## 研究背景与动机

工具学习（Tool Learning）旨在为 LLM 配备外部工具以增强其解决实际任务的能力。在实际应用中，由于 LLM 上下文窗口有限，从大规模工具集中检索有用工具是关键的第一步。然而，现有的工具使用基准大多通过人工预标注 10-20 个相关工具来简化这一检索过程，与真实场景相距甚远。

作者通过先导实验发现了两个关键问题：
1. 当用检索到的工具替代官方标注的工具集时，agent 的任务通过率大幅下降
2. 即使是 ColBERTv2 这样的强力检索模型，在工具检索任务上也表现不佳

这说明工具检索与传统信息检索存在根本性差异：工具检索要求理解查询意图与工具功能之间的对应关系，且完成一个任务通常需要多个工具的组合，这比传统的文档检索更具挑战性。

## 方法详解

### 整体框架

ToolRet 的构建遵循三个核心流程：数据收集 → 数据采样 → 指令构建，最终形成一个包含 7,615 个检索任务和 43,215 个工具的综合评测基准。

### 关键设计

1. **多源数据收集**: 从三类来源系统收集数据——(i) ACL/NeurIPS 等 AI 顶会发表的工具使用基准；(ii) EMNLP/CIKM 等会议的资源数据集；(iii) HuggingFace 等开源平台的高质量数据集。收集了 2023.8-2024.12 期间的 30+ 数据集，并统一为与 MTEB/BEIR 对齐的检索格式（查询 + 目标工具标签）。这样做的动机是确保评测的多样性和全面性，覆盖不同类型的工具文档、领域和查询长度。

2. **智能数据采样**: 针对数据集间的规模不平衡问题，采用基于 NV-embed-v1 嵌入的 K-means 聚类进行任务采样，每个聚类随机抽取一个任务。工具集方面则通过人工审查合并重叠的工具集（如 COLT 与 ToolBench 的工具集）。这种设计既消除了冗余，又保持了评测的代表性。

3. **目标感知指令构建**: 为支持指令式检索评估，设计了一种基于 GPT-4o 的自动指令生成策略。首先邀请 3 位 NLP/IR 背景的专家手工编写 100 条种子指令，然后利用 GPT-4o 通过上下文学习为每个任务生成指令。随机从已生成和手工指令池中抽取示例，以增强多样性。

4. **大规模训练数据 ToolRet-train**: 扩展数据收集流程至 ToolACE、APIGen 和 ToolBench 的训练集，构建了超过 200k 检索任务的训练数据。每个训练样本包含查询、生成的指令、目标工具和由 NV-embed-v1 检索得到的 10 个负例工具。

### 工具语料库统计

最终语料库包含三类工具：Web API（36,978 个）、代码函数（3,794 个）和自定义应用（2,443 个），查询/指令平均长度分别为 46.87/43.43 token，工具文档平均长度为 174.56 token。

## 实验关键数据

### 主实验

作者在 ToolRet 上系统评估了六类 IR 模型：

| 模型类型 | 代表模型 | nDCG@10 | 说明 |
|----------|----------|---------|------|
| 最佳嵌入模型 | NV-embed-v1 | 33.83 | 传统 IR 基准的 SOTA |
| ColBERTv2 | ColBERTv2 | - | 强力检索器，仍表现不佳 |
| 训练后模型 | NV-embed-v1 + ToolRet-train | 显著提升 | 训练后性能大幅改善 |

### 关键发现

1. **检索质量与端到端性能强相关**: 工具检索的 Recall@10 与工具使用 agent 的任务通过率呈正相关（如图 1 所示），证明工具检索质量直接影响 LLM agent 的最终表现。

2. **性能差距的两大原因**:
   - 查询与目标工具之间的词汇重叠度低，要求 IR 模型具备更强的语义表示能力
   - 从传统信息检索到工具检索存在任务迁移，未经专门优化的 IR 模型表现次优

3. **训练数据的显著效果**: 在 ToolRet-train 上训练后的 IR 模型表现出显著的检索性能提升，且在集成到工具使用 LLM 后，端到端任务通过率也相应提高。

### 消融实验

| 配置 | 关键发现 | 说明 |
|------|----------|------|
| 带指令 vs 不带指令 | 指令能提升检索性能 | 支持指令式检索的设计 |
| 不同工具类型 | Web API 检索更难 | 因工具数量更大且功能相似度高 |
| 不同负例策略 | 检索负例优于随机负例 | 硬负例训练更有效 |

## 亮点与洞察

- **首次系统化评测工具检索**：填补了工具检索作为独立任务的评测空白，揭示了传统 IR 模型在该场景下的不足。
- **端到端视角**：不仅评估检索本身，还分析了检索质量对 agent 最终任务表现的影响，具有很强的实用指导意义。
- **目标感知指令生成**：巧妙结合人工种子指令和 LLM 自动生成，既保证了质量又实现了规模化。这种"人机协作"的标注范式对其他需要大规模指令数据的任务也有借鉴价值。

## 局限性 / 可改进方向

- 工具文档质量参差不齐：来自不同源的工具文档格式和信息丰富度差异大，可能影响检索模型的公平评测。
- 仅考虑了工具作为"flat document"的检索范式，未探索工具间的组合和依赖关系在检索阶段的建模。
- 训练数据中的负例挖掘策略较为简单（仅使用 NV-embed-v1 的检索负例），更复杂的硬负例策略有望进一步提升性能。
- 尚未充分探索多粒度工具表示和层次化检索策略。

## 相关工作与启发

- 与 MTEB/BEIR 等通用检索基准互补，ToolRet 专注于工具检索这一垂直领域。
- 工具检索可看作一种特殊的"语义匹配"任务，但其挑战在于需要理解"用户意图 → 工具功能"的映射关系，而非简单的查询-文档匹配。
- 启发：随着 LLM Agent 生态的发展，工具检索将成为越来越关键的基础能力，值得投入更多研究。

## 补充说明

ToolRet 的数据收集覆盖了 2023.8 至 2024.12 期间发布的 34 个数据集，最终合并去重后形成 43,215 个工具的统一语料库，其中 Web API 占比最大（85.6%），代码函数和自定义应用分别占 8.8% 和 5.6%。基准的多样性体现在：工具文档风格差异大（从简洁的 API 描述到详细的代码注释），查询长度从单句指令到多步骤任务描述不等。这种多样性使得 ToolRet 比任何单一来源的工具基准都更接近真实的工具检索场景。
