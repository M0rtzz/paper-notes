# What Makes a Good Natural Language Prompt?

**会议**: ACL 2025  
**arXiv**: [2506.06950](https://arxiv.org/abs/2506.06950)  
**代码**: 无  
**领域**: LLM / 提示学习  
**关键词**: prompt质量, 元分析, 属性框架, prompt工程, 以人为中心

## 一句话总结

通过元分析150+篇prompting文献和博客，提出以属性为中心、以人为中心的prompt质量评估框架（6个维度21个属性），实验发现单属性增强往往比多属性组合更有效，且不同属性在不同任务/模型上效果差异大。

## 研究背景与动机

**领域现状**: prompt工程已成为与LLM交互的核心技能，但"什么是好的prompt"缺乏系统化理解。**现有痛点**: 现有研究零散地提出各种prompting技术（CoT/few-shot/角色扮演等），但缺乏统一框架理解它们为什么有效、何时有效。**核心矛盾**: prompt质量是多维的，但现有评估通常只看最终任务性能，不分析prompt的哪些属性起了作用。**本文要解决什么**: 构建系统化的prompt属性分类框架并实验验证哪些属性最重要。**切入角度**: 从大量文献中提取prompt设计的共性模式，归纳为可操作的属性。**核心idea**: prompt质量可分解为6个维度21个可独立评估的属性。

## 方法详解

### 整体框架

文献元分析→提取prompt属性→构建6×21分类框架→实验验证各属性和组合的效果。

### 关键设计

1. **6维度21属性框架**:
    - 功能：将prompt质量分解为6个维度（清晰性/特异性/上下文/推理/格式/元认知）下的21个可操作属性
    - 核心思路：从150+文献和实践中提取共性，每个属性有明确定义和增强方法（如"清晰性-简洁""推理-CoT""格式-结构化输出"）
    - 设计动机：此前的prompt技术是"技巧集合"而非"系统框架"——21属性使prompt设计从经验变为工程

2. **单属性vs多属性组合实验**:
    - 功能：系统测试每个属性的单独效果和多属性组合效果
    - 核心思路：对基线prompt逐一添加单个属性增强，然后测试2-5个属性的组合，比较性能变化
    - 设计动机：直觉认为"属性越多越好"——但实验发现组合反而可能降低性能（属性间可能冲突）

3. **以人为中心的评估**:
    - 功能：除自动指标外加入人类评估维度
    - 核心思路：评估prompt的可读性、意图传达清晰度等用户体验维度
    - 设计动机：好的prompt不仅要让LLM表现好，还要让人能理解和维护

### 损失函数 / 训练策略

不涉及训练。实验在多个LLM上零/少样本评估。

## 实验关键数据

### 主实验

不同属性对任务性能的影响（跨模型平均）：

| 属性维度 | 代表属性 | 平均提升 | 最佳场景 |
|---------|---------|:---:|------|
| 推理 | Chain-of-Thought | +8-15% | 数学/逻辑推理 |
| 上下文 | Few-shot示例 | +5-12% | 分类/生成 |
| 特异性 | 角色定义 | +3-8% | 创意写作 |
| 格式 | 结构化输出 | +2-5% | 信息提取 |

### 消融实验

单属性 vs 多属性组合：

| 配置 | 平均性能 | 说明 |
|------|:---:|------|
| 基线prompt | 100% | — |
| +最佳单属性 | **110-115%** | 显著提升 |
| +2-3属性组合 | 108-112% | 不总是优于单属性 |
| +5属性组合 | 105-110% | 属性冲突可能导致退化 |

### 关键发现

1. **单属性增强往往比多属性组合更有效**: 精准添加一个合适属性>盲目堆砌多个属性
2. **不同模型对同一属性响应差异大**: CoT对GPT-4有效但对小模型可能无效
3. **属性间可能冲突**: 如"简洁"与"详细上下文"矛盾
4. **推理和上下文是最普遍有效的维度**

## 亮点与洞察

- **从零散技巧到系统框架**: 21属性框架使prompt设计可工程化
- **"少即是多"的反直觉发现**: 堆砌属性不如精准匹配
- **实用指导**: 根据任务类型选择最相关的1-2个属性增强
- **以人为中心的视角**: 好prompt需要对人和LLM双向友好

## 局限性 / 可改进方向

- 文献覆盖截至发表时，新技术（如reflection/planning）未纳入
- 21个属性的边界可能模糊
- 属性组合的搜索空间巨大，实验未穷尽
- 不同语言下的属性效果未验证

## 相关工作与启发

- **vs prompt自动优化（APE/OPRO）**: 自动搜索——本文提供人类可理解的设计框架
- **vs prompt工程指南（OpenAI/Anthropic）**: 实践经验——本文提供系统化研究支撑
- **启发**: prompt设计应从"试错"转向"属性匹配"——先诊断任务需要什么属性再针对性增强
