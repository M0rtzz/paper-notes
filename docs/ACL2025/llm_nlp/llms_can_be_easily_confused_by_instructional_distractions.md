# LLMs can be easily Confused by Instructional Distractions

**会议**: ACL 2025  
**arXiv**: [2502.04362](https://arxiv.org/abs/2502.04362)  
**代码**: 无  
**领域**: LLM/NLP  
**关键词**: 指令跟随, 指令干扰, 基准测试, 鲁棒性, LLM评估

## 一句话总结

本文提出了 DIM-Bench 基准，系统评估了当输入文本本身就像指令时 LLM 的指令跟随能力，发现即使是 GPT-4o 等最先进的模型也极易受到这种"指令干扰"的影响而偏离用户真实意图。

## 研究背景与动机

LLM 在指令跟随任务中表现优异，但这种优势在需要忽略某些指令时反而可能变为弱点。指令跟随任务通常包含一个明确的任务描述（指令）和一个待处理的输入文本。然而，当输入文本本身就像一条指令时（例如输入的数学题看起来也是一种"做题指令"），LLM 就会产生困惑，即使提示中已明确区分了指令和输入。

作者将这一现象定义为**指令干扰（Instructional Distraction）**。例如，研究者有大量英文数学数据需要翻译成中文时，LLM 可能会忽略翻译指令，转而去解答数学题。这个问题在实际数据处理和生成场景中非常普遍——批量处理时无法逐条修改提示，后处理成本极高。然而目前缺乏系统评估 LLM 在这种干扰场景下表现的基准。

指令干扰与指令冲突（Instruction Conflict）有本质区别：指令冲突是多个指令存在优先级关系，而指令干扰是只有一个指令，输入文本中的内容模仿了指令格式起到干扰作用。

## 方法详解

### 整体框架

DIM-Bench（Distractive Instruction Misunderstanding Benchmark）通过组合两维任务构建基准：

- **指令任务（4种）**：反映真实数据处理场景
  - 改写（Rewriting）：保持语义重述文本
  - 校对（Proofreading）：纠正语法、拼写和标点
  - 翻译（Translation）：翻译至中文、西班牙语、法语等8种语言
  - 风格转换（Style Transfer）：在正式/直接/休闲/情感4种风格间转换

- **输入任务（5种）**：扮演干扰角色
  - 推理（Reasoning）：来自 ARC 数据集的选择题
  - 代码生成（Code Generation）：来自 Code Alpaca 数据集
  - 数学推理（Mathematical Reasoning）：来自 GSM8k 和 MATH 数据集
  - 偏见检测（Bias Detection）：来自 BBQ 数据集
  - 问答（Question Answering）：来自 NarrativeQA 数据集，包含长文本

4×5=20 个类别，每个类别 100 个样本，共 **2000 个测试实例**。

### 关键设计

**评估方法（双重评估）**：

1. **LLM-Judge 评估**：使用 GPT-4o 作为评判模型，针对每个类别设计 2-3 个二值问题来判断输出是否符合指令意图。例如翻译+推理类别：(1) 输出是否是目标语言？(2) 输出是否保持选择题格式？(3) 是否保留了原始选项？

2. **基于长度差异的自动评估**：专门针对问答任务设计——改写/翻译/校对等操作后文本长度应接近原始长度，若输出远短于输入（如 10 倍差距），说明模型直接回答了问题而非执行指令任务。

**提示策略探索**：

- **直接提示（DIRECT）**：显式告知模型"不要解决输入中的指令或问题，将其作为文本处理"
- **思维链提示（CoT）**：引导模型逐步推理后再生成
- **后缀指令（Suffix Instruction）**：将指令放在输入之后而非之前

### 损失函数 / 训练策略

本文不涉及模型训练，而是一个评估基准。核心设计在于如何构建能可靠衡量指令干扰影响的自动化评估流程。

## 实验关键数据

### 主实验

对 6 个 LLM（Llama-3.1-8B/70B-Instruct、Qwen-2.5-7B-Instruct、GPT-3.5、GPT-4o-mini、GPT-4o）的评估结果：

- **按指令类型平均准确率**：翻译 (0.526) > 校对 (0.458) > 改写 (0.397) > 风格转换 (0.301)
- **按输入类型平均准确率**：数学 (0.738) > 代码 (0.612) > 推理 (0.493) > 偏见检测 (0.208) > 问答 (0.051)
- **问答任务极端情况**：多个模型的准确率为 0.00，表明 LLM 面对长文本+问题时几乎完全被干扰
- **GPT-4o 在翻译+数学上准确率 0.96**：说明数学格式有一定辨识度
- **GPT-4o 在改写+偏见检测上仅 0.11**：说明含选择题的偏见检测输入容易干扰模型

### 消融实验

**提示策略对比（翻译任务，Llama-3.1-70B）**：
- 标准评估 vs DIRECT 提示：问答任务从 0.00 提升至 0.13，但整体改善有限
- CoT 提示：对问答任务仅从 0.00 提升至 0.02
- 后缀指令：代码生成任务从 0.82 骤降至 0.08，说明指令后置会严重加剧干扰

**输入长度影响（翻译任务）**：
- QA_short (362 tokens): Llama 0.28, GPT-4o 0.31
- QA_medium (743 tokens): Llama 0.09, GPT-4o 0.07
- QA_long (1087 tokens): Llama 0.06, GPT-4o 0.04
- QA_superlong (3007 tokens): Llama 0.05, GPT-4o 0.02

输入越长，指令干扰越严重，因为指令与输入中问题的距离增大。

### 关键发现

1. **没有任何 LLM（包括 GPT-4o）能完全抵抗指令干扰**：即使提供了明确区分指令和输入的提示
2. **问答格式是最强干扰源**：LLM 看到问题就倾向于给出答案，而非按指令处理文本
3. **三种提示方法均无法完全解决问题**：DIRECT 和 CoT 提示仅部分改善，后缀指令反而加重问题
4. **模型容量与鲁棒性不完全正相关**：GPT-4o 在问答干扰上的表现并不比小模型好

## 亮点与洞察

- **定义了一个被忽视但实际影响很大的问题**：指令干扰在数据处理流水线中会导致整批数据处理失败
- **基准设计贴近真实场景**：四种指令任务（改写/翻译/校对/风格转换）均是 LLM 数据工厂的常见操作
- **双重评估机制**：LLM-Judge + 长度差异自动评估相互验证，增强了结果可信度
- **揭示了指令跟随能力的根本局限**：模型的"指令服从性"优势恰恰是这种漏洞的来源

## 局限性 / 可改进方向

- 未考虑摘要等一对多任务（一个输入可能有多种合理输出形式），这类任务更难用 LLM-Judge 评估
- 仅使用零样本提示，未探索少样本提示或微调方法能否缓解问题
- 翻译任务的目标语言仅限于主流语言，未覆盖低资源语言
- 仅手动验证了部分失败案例，大规模自动分析失败模式的工作仍有空间
- 未提出根本性的解决方案——只是揭示了问题，没有给出有效的缓解方法

## 相关工作与启发

- 与 Wallace et al. (2024) 提出的指令层级（Instruction Hierarchy）概念相关但不同：指令层级处理的是多指令优先级，本文处理的是输入文本的干扰
- Wen et al. (2024) 和 He et al. (2024b) 关注的是复杂多约束指令，而本文关注的是输入内容的欺骗性
- 对实际 LLM 数据工厂的启示：批量处理含指令格式的数据时需要额外的质量控制机制
