# APEX-MEM: Agentic Semi-Structured Memory with Temporal Reasoning for Long-Term Conversational AI

**会议**: ACL 2026  
**arXiv**: [2604.14362](https://arxiv.org/abs/2604.14362)  
**代码**: 无  
**领域**: Agent / 对话系统  
**关键词**: 长期记忆, 属性图, 时序推理, 对话AI, 多工具检索

## 一句话总结
APEX-MEM 提出了一种基于属性图（Property Graph）+ 只追加式事件存储 + 多工具检索代理的对话记忆系统，通过领域无关本体和检索时时序推理，在 LOCOMO 和 LongMemEval 上分别达到 88.88% 和 86.2% 的准确率，显著超越现有结构化记忆方法。

## 研究背景与动机

**领域现状**：LLM 在长期对话中的记忆能力仍然薄弱。简单扩大上下文窗口会引入噪声和幻觉，而基于 RAG 的检索方法将记忆视为非结构化文本，无法维护规范化实体、追踪事实演变或区分持久信息与临时对话内容。

**现有痛点**：现有结构化记忆系统如 Mem0 虽然引入了实体中心图，但实体类别有限，信息主要以实体间关系存储，难以捕捉细粒度属性和事实的时序演变。此外，多数系统在写入时就合并或覆盖旧信息，存在丢失时序推理所需上下文细节的风险。

**核心矛盾**：记忆系统需要在"保留完整历史"和"减少检索噪声"之间取得平衡——过早合并会丢信息，完整保留又会增加噪声。

**本文目标**：构建一个既能保留完整对话历史演变，又能在检索时动态解析冲突和过时信息的记忆框架。

**切入角度**：借鉴知识图谱中的事件本体思想，将对话事实锚定到时间戳事件上而非直接挂在实体上，从而实现追加写入 + 检索时解析的设计范式。

**核心 idea**：用"属性图 + 只追加事件存储 + 检索时时序解析"三位一体的架构，替代传统的"写入时合并"记忆模式。

## 方法详解

### 整体框架
系统分为两大部分：图构建（Graph Construction）和图查询（Graph QA Agent）。输入是多轮对话文本，输出是针对用户问题的记忆感知回答。对话文本经过事实抽取、实体解析后写入属性图，查询时由 ReAct 风格的代理调用多种工具进行检索和推理。

### 关键设计

1. **领域无关时序事件本体**:

    - 功能：为对话记忆提供统一的结构化表示
    - 核心思路：定义了 35 类实体类型（Person、Organization、Product 等），事实表示为时序锚定的"主体-属性-值"三元组 $f=(s,p,v,\delta,[t_{from},t_{to}],c,\mathcal{E})$，每个事实都锚定到带时间戳的对话事件上。事件作为一等公民建模，包含类型、时间、地点、参与者和关联事实。
    - 设计动机：与 Mem0 等仅以实体关系建模不同，事件本体能同时捕捉实体属性和时序演变，支持"同一实体在不同时间有不同属性值"的场景

2. **只追加式事件存储（Append-Only Storage）**:

    - 功能：保留信息的完整时序演变历史
    - 核心思路：新事实不覆盖旧事实，而是作为新事件节点追加到图中。每条事实都带有时间有效区间 $[t_{from}, t_{to}]$ 和置信度 $c \in [0,1]$。冲突和修订信息共存于图中。
    - 设计动机：避免写入时过早合并导致的信息丢失，将冲突解析推迟到检索时，根据查询上下文动态决定哪条信息有效

3. **多工具检索代理**:

    - 功能：在查询时从属性图中检索并推理出答案
    - 核心思路：ReAct 风格代理配备四种工具——SchemaViewer（元级规划指导）、EntityLookup（实体检索和规范化）、GraphSQL（基于 SQL 的结构化图遍历，支持时序排序、区间计算、多跳推理）、Search（混合语义+词法搜索）。代理交替执行推理和工具调用，逐步构建答案。
    - 设计动机：不同类型的问题需要不同的检索策略——实体查询用 EntityLookup，时序计算用 GraphSQL，开放域问题用 Search，元规划用 SchemaViewer

### 损失函数 / 训练策略
图构建阶段使用 few-shot prompted LLM 进行事实抽取，实体解析采用"密集语义检索 + 结构化 LLM 推理"的管线。在线构建模式下，对超长对话（>1000 文档）先用语义和词法搜索筛选相关文档子集，再构建局部图。

## 实验关键数据

### 主实验

| 数据集 | 方法 | 整体准确率 | 单跳 | 多跳 | 时序 | 开放域 |
|--------|------|-----------|------|------|------|--------|
| LOCOMO | APEX-MEM (GPT5) | **88.88%** | 89.88% | 86.29% | 90.63% | 91.68% |
| LOCOMO | MIRIX | 85.38% | 85.11% | 83.70% | 65.62% | 88.39% |
| LOCOMO | Mem0 | 68.44% | 65.71% | 47.19% | 75.71% | 58.13% |
| LOCOMO | Full Context GPT4o | 87.52% | 88.53% | 77.70% | 71.88% | 92.70% |

### 消融实验

| 配置 | 关键表现 | 说明 |
|------|---------|------|
| 完整 APEX-MEM | 88.88% | 四种工具协同 |
| 仅 SimpleSearch | 77.90% | 去掉结构化图后大幅下降 |
| 仅 EntityLookup | - | 实体中心检索，适合单跳 |
| 仅 GraphSQL | - | 结构化查询，适合时序推理 |

### 关键发现
- APEX-MEM 在时序推理任务上优势最大（90.63% vs MIRIX 65.62%），证明事件本体+只追加存储对时序推理的价值
- 即使使用较弱的 LLM（Claude 4.5 Haiku），APEX-MEM 仍达 84.25%，说明架构设计本身的贡献大于底层 LLM 能力
- 在对抗性问题上达 86.77%，远超 Full Context 方法，结构化记忆有效抑制了噪声干扰

## 亮点与洞察
- **只追加+检索时解析**的设计范式非常优雅：写入时不做任何决策，所有智能都放在读取端，这与"write-ahead log"在数据库中的理念异曲同工
- 用 SQLite 作为图存储后端，使得 GraphSQL 工具可以直接写 SQL 做时序计算，比自定义图查询语言更通用
- 35 类实体本体虽然看似复杂，但通过领域无关设计实现了跨领域适用，避免了针对每个场景重新设计 schema

## 局限与展望
- 图构建依赖 LLM 的事实抽取和实体解析，抽取质量受 LLM 能力限制
- 只追加存储会导致图不断膨胀，长期运行的存储和检索效率有待验证
- 评估仅在 LOCOMO 和 LongMemEval 上进行，缺乏真实部署场景的验证

## 相关工作与启发
- **vs Mem0/Mem0g**: Mem0 实体类别有限且直接在实体上存储关系，APEX-MEM 引入事件本体和只追加存储，时序推理能力显著更强
- **vs MIRIX**: MIRIX 用六个专用记忆存储+多代理路由达到 85.4%，但急切更新策略可能丢失时序信息，APEX-MEM 用更简洁的架构达到更高准确率
- **vs Zep**: Zep 构建时序感知知识图谱但严重依赖文本检索，APEX-MEM 的 GraphSQL 工具提供了更精确的结构化查询能力
