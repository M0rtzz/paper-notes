# BlazeFL: Fast and Deterministic Federated Learning Simulation

**会议**: CVPR 2026 (Workshop: FedVision)  
**arXiv**: [2604.03606](https://arxiv.org/abs/2604.03606)  
**代码**: [GitHub](https://github.com/kitsuyaazuma/blazefl) (有)  
**领域**: 联邦学习 / 系统优化  
**关键词**: 联邦学习仿真, 确定性可复现, 自由线程, 共享内存, 轻量框架

## 一句话总结
提出 BlazeFL 联邦学习仿真框架，利用 Python 自由线程（free-threading）实现共享内存通信和客户端隔离随机数管理，在通信密集型工作负载上实现最高 3.1× 加速，且保证比特级可复现。

## 研究背景与动机
**领域现状**: 联邦学习（FL）研究大量依赖单机仿真（数百到数千虚拟客户端），Flower、FedML 等框架为主流工具。

**现有痛点**: (a) 并行客户端训练引入不确定性——共享随机状态、调度变异导致结果不可复现；(b) 多进程模式下参数序列化和进程间通信（IPC）开销大，尤其在通信密集型场景。

**核心矛盾**: 提高吞吐量与保持可复现性之间存在权衡——并行越多，非确定性越强；研究者要么牺牲速度要么牺牲可复现性。

**本文要解决**: 设计一个既快又确定性的单机 FL 仿真框架。

**切入角度**: 利用 Python 3.14 的自由线程（PEP 703/779）绕过 GIL，用线程替代进程实现真正并行，通过共享内存避免序列化开销。

**核心idea**: 线程级共享内存执行 + 客户端隔离 RNG 流 = 快速且比特级可复现的 FL 仿真。

## 方法详解

### 整体框架
主线程协调客户端调度 → 工作线程在共享地址空间中执行 → 服务器参数直接通过共享内存广播 → 客户端结果按固定顺序收集 → 每个客户端绑定独立 RNG 流。

### 关键设计

1. **共享内存执行（Free-Threading）**:

    - 客户端作为工作线程在单进程内执行，模型参数通过共享内存直接访问
    - 无需跨进程序列化、IPC、外部对象存储
    - 也提供基于进程的共享内存模式作为对照
    - **设计动机**: GIL 限制了传统 Python 线程的 CPU 并行性，多进程引入序列化开销；自由线程突破了这一瓶颈

2. **确定性随机数管理**:

    - 每个客户端绑定独立 RNG 套件（从确定性种子计划初始化）
    - 客户端本地的随机操作（采样、打乱、增强、dropout）与工作调度解耦
    - 结果按采样顺序（而非完成顺序）收集，确保浮点累加顺序一致
    - **设计动机**: FL 仿真中非确定性来自两个源头——(a) 共享/不一致的随机状态，(b) 浮点加法的完成顺序依赖（浮点加法不满足结合律）

3. **协议式接口（Protocol-Based）**:

    - 使用 Python `typing.Protocol` 定义接口，无需继承特定基类
    - 普通 PyTorch 训练组件可最小修改接入
    - **设计动机**: 降低框架锁定，让实验代码尽可能贴近标准 PyTorch 训练循环

### 损失函数 / 训练策略
- 采用标准 FedAvg 聚合，100 个客户端，Non-IID 分区（每客户端 2 类）
- 每轮 5 epochs 本地训练，500 样本/客户端，10,000 测试样本全局评估

## 实验关键数据

### 主实验（墙钟时间，5 轮通信）
| 模型 | 并行度 | BlazeFL (FT) | BlazeFL (Proc) | Flower (Ray) | 加速 |
|------|--------|-------------|---------------|-------------|------|
| CNN | P=32 | **最快** | 中等 | 最慢 | 最高 3.1× |
| ResNet-18 | P=32 | **最快** | 中等 | 较慢 | ~1.4× |
| ResNet-50 | P=16 | **最快** | 较快 | 相当 | ~1.1× |
| ResNet-101 | P=8 | 相当 | 相当 | 相当 | 接近 |

### 消融实验（可复现性分析，P=32，10 次运行）
| 配置 | 最终精度标准差 | 轮次级 SHA-256 一致 | 说明 |
|------|--------------|-------------------|------|
| Flower (无种子) | 1.24 pp | ❌ | 完全不可复现 |
| Flower (全局种子) | 0.18 pp | ❌ | 仍有微小差异 |
| BlazeFL (进程模式) | **0.00** | ✅ | 比特级一致 |
| BlazeFL (自由线程) | **0.00** | ✅ | 比特级一致 |

### 关键发现
- 在通信密集型场景（轻量 CNN），共享内存优势最大（3.1× 加速）
- 随着模型增大，计算占主导，通信优化收益递减
- BlazeFL 在不同并行度 P∈{1,2,4,...,64} 下均产生比特级相同结果
- 诊断发现 Flower 的非确定性源于聚合时浮点累加顺序变化，初始 $10^{-6}$ 级误差逐轮放大

## 亮点与洞察
- 首个利用 Python 自由线程进行 FL 仿真的框架，展示了新 Python 特性的实际价值
- "确定性可复现"不仅是工程优势，更对 FL 算法的细粒度行为分析至关重要
- Flower 非确定性的诊断分析很有教育意义——即使全局设种子也不够，必须单独管理每客户端 RNG
- 轻量依赖（仅 Python 标准库 + PyTorch）提升了实验的可移植性

## 局限性 / 可改进方向
- 仅支持单机仿真，不适用于真正的分布式 FL 部署
- VRAM 有限时，自由线程的 CUDA 分配器可能成为瓶颈（全局互斥锁）
- Python 自由线程生态尚未成熟，第三方库兼容性有限
- 视觉增强管道中某些操作可能依赖全局 RNG 状态，需用户手动适配

## 相关工作与启发
- Flower + Ray 是广泛使用的 FL 框架，但跨进程开销在单机场景下不可忽视
- PyTorch 共享内存张量可减少序列化开销，但仍需进程管理
- 启示：对于单机 FL 实验，"轻量 + 确定性"比"功能全面 + 可扩展"更实用
